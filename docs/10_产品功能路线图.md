# CodeCoach AI 产品功能路线图

## 1. 文档说明

本文档用于规划 CodeCoach AI 从 MVP 验证版到完整产品形态的功能演进路线。

当前项目已经完成 MVP 阶段开发，验证了核心业务闭环的可行性：

```text
用户注册登录
-> 创建项目档案
-> 发起 AI 项目拷打训练
-> 多轮回答与追问
-> 生成训练报告
-> 查看训练历史
```

接下来项目进入产品化规划与能力增强阶段。

本阶段目标不是继续堆叠零散功能，而是围绕完整产品形态，分阶段建设 AI 能力底座、八股问答训练、知识学习闭环、能力画像、RAG 个性化增强和最终部署上线能力。

---

## 2. 产品定位

CodeCoach AI 的长期产品定位是：

```text
面向 Java 后端求职者的 AI 面试训练与学习工作台。
```

它不是单纯的 AI 聊天工具，也不是普通题库系统，而是一个围绕“求职训练”构建的长期学习产品。

核心价值包括：

1. 帮助用户沉淀项目经历；
2. 基于项目档案进行 AI 项目拷打；
3. 基于知识点进行八股问答训练；
4. 基于训练结果生成报告和薄弱点分析；
5. 基于薄弱点推荐知识学习和专项训练；
6. 后续结合简历、项目文档和学习笔记进行 RAG 个性化追问；
7. 长期沉淀用户能力画像和成长趋势。

## 3. 当前阶段：V1.0 MVP 验证版

### 3.1 阶段目标

V1.0 的目标是验证 CodeCoach AI 的核心产品闭环是否成立。

核心问题：


## 5. V2.0：AI 能力工程化版

### 5.1 阶段目标

V2.0 的目标是将当前简单的大模型 API 调用升级为可维护、可观测、可扩展的 AI 能力底座。

当前问题：

1. Prompt 可能散落在 Java 代码中；
2. Prompt 结构和版本不可管理；
3. 模型 JSON 输出不稳定；
4. AI 调用日志能力不足；
5. 模型切换能力有限；
6. AI 错误处理和可观测性不够完善；
7. 不同难度、不同训练风格的 Prompt 差异化不足。

---

### 5.2 核心能力

#### 5.2.1 Prompt 模板资源化

将 Prompt 从 Java 代码中抽离，形成资源化模板。

建议目录：

```text
backend/src/main/resources/prompts
├── interview_first_question.md
├── interview_feedback_next_question.md
└── interview_report.md
```

目标：

1. Prompt 可维护；
2. Prompt 可版本化；
3. Prompt 支持变量注入；
4. Prompt 调整不需要改业务代码；
5. 后续八股训练、知识学习、RAG 都可复用模板体系。

---

#### 5.2.2 Prompt 上下文编排

将训练上下文结构化注入 Prompt。

上下文包括：

1. 项目名称；
2. 项目描述；
3. 技术栈；
4. 负责模块；
5. 项目亮点；
6. 项目难点；
7. 目标岗位；
8. 难度；
9. 当前轮次；
10. 历史问答；
11. 用户本轮回答。

目标：

```text
不是简单把用户回答转发给模型，而是构造面向 Java 后端面试场景的结构化上下文。
```

---

#### 5.2.3 结构化输出解析

对模型返回的 JSON 结果进行稳定解析。

需要处理：

1. 模型返回 Markdown 代码块；
2. 模型返回 JSON 前后带解释文字；
3. JSON 字段缺失；
4. JSON 格式不合法；
5. totalScore 类型异常；
6. 数组字段为空或不存在；
7. qaReview 结构异常。

建议新增：

```text
AiJsonParser
AiResponseValidator
```

原则：

1. 不 fallback 到 Mock；
2. 解析失败返回明确错误；
3. 记录原始输出，便于排查；
4. 保证前端不会因为模型输出异常而崩溃。

---

#### 5.2.4 AI 调用日志

完善 `ai_call_log` 的使用。

建议记录：

1. userId；
2. sessionId；
3. provider；
4. modelName；
5. requestType；
6. latencyMs；
7. success；
8. errorMessage；
9. createdAt；
10. rawResponse，可选；
11. promptTokens / completionTokens / totalTokens，若模型返回。

调用类型示例：

```text
FIRST_QUESTION
FEEDBACK_NEXT_QUESTION
REPORT_GENERATION
KNOWLEDGE_QUESTION
KNOWLEDGE_REPORT
```

价值：

1. 排查 AI 调用失败；
2. 统计模型耗时；
3. 观察不同任务的稳定性；
4. 后续分析 token 成本；
5. 支撑生产环境监控。

---

#### 5.2.5 多模型适配

当前优先使用 OpenAI-Compatible API。

目标：

```text
业务层不依赖具体模型厂商。
```

建议结构：

```text
AiInterviewService
├── OpenAiCompatibleAiInterviewServiceImpl
└── MockAiInterviewServiceImpl，仅测试 / mock profile 使用
```

配置示例：

```yaml
ai:
  provider: openai-compatible
  openai-compatible:
    base-url: ${AI_BASE_URL}
    api-key: ${AI_API_KEY}
    model: ${AI_MODEL}
    timeout-seconds: 120
```

要求：

1. API Key 不写死；
2. API Key 不提交 Git；
3. 模型可通过配置切换；
4. 线上失败不 fallback Mock；
5. Mock 只用于本地测试。

---

#### 5.2.6 训练难度差异化

当前 difficulty 支持：

```text
EASY
NORMAL
HARD
```

后续需要在 Prompt 中体现差异。

示例：

| 难度 | 训练风格 |
|---|---|
| EASY | 更引导，问题更基础，反馈更友好 |
| NORMAL | 模拟常规实习 / 校招面试 |
| HARD | 深挖技术细节、异常场景、性能和架构设计 |

## 9. V3.0：RAG 个性化增强版

### 9.1 阶段目标

V3.0 的目标是基于用户简历、项目文档和学习笔记，生成更个性化的训练内容。

不是做通用文档问答，而是服务于：

```text
更精准的面试追问。
```

---

### 9.2 典型场景

用户上传：

1. 简历 PDF；
2. 项目 README；
3. 项目设计文档；
4. 学习笔记；
5. 面试复盘。

系统处理：

```text
文件上传
-> 文档解析
-> 文本切片
-> Embedding
-> 向量存储
-> 检索相关上下文
-> 拼接 Prompt
-> 生成个性化追问
```

---

### 9.3 技术建议

轻量方案：

| 能力 | 建议 |
|---|---|
| 文件存储 | 本地文件 / MinIO |
| 文档解析 | PDFBox / Apache Tika |
| 向量库 | PostgreSQL + pgvector / Qdrant |
| Embedding | 外部 API |
| 检索 | TopK 相似度检索 |
| 权限 | 按 userId 隔离 |

不建议第一版直接上 Milvus，部署复杂度偏高。

---

### 9.4 推荐任务拆分

| 编号 | 任务 | 优先级 |
