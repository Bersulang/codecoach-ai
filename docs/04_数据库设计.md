# CodeCoach AI 数据库设计

## 1. 文档说明

本文档用于描述 CodeCoach AI MVP 版本的数据库表结构设计。

MVP 版本核心业务闭环为：

用户创建项目档案 -> 发起项目拷打训练 -> AI 生成问题 -> 用户回答 -> AI 反馈和追问 -> 用户结束训练 -> 系统生成训练报告。

本文档基于《03_领域模型设计.md》中的领域对象进行数据库落表设计。

---

## 2. 数据库选型

MVP 阶段使用 MySQL 作为主业务数据库。

原因：

1. 项目第一版主要是用户、项目、训练会话、训练消息、训练报告等结构化数据；
2. MySQL 适合存储核心业务数据；
3. Java 后端实习面试中 MySQL 是高频考察点；
4. 后续可以在 RAG 版本中引入 PostgreSQL + pgvector 或其他向量数据库。

## 5. user 用户表

## 5.1 表说明

`user` 表用于存储系统注册用户信息。

MVP 阶段主要用于登录认证和用户数据隔离。

## 5.2 建表 SQL

```sql
CREATE TABLE user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '用户ID',
    username VARCHAR(64) NOT NULL COMMENT '用户名',
    password VARCHAR(255) NOT NULL COMMENT '加密后的密码',
    nickname VARCHAR(64) DEFAULT NULL COMMENT '用户昵称',
    avatar_url VARCHAR(512) DEFAULT NULL COMMENT '头像地址',
    role VARCHAR(32) NOT NULL DEFAULT 'USER' COMMENT '角色：USER/ADMIN',
    status TINYINT NOT NULL DEFAULT 1 COMMENT '状态：1正常，0禁用',
    is_deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除：0未删除，1已删除',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    UNIQUE KEY uk_username (username),
    KEY idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

## 5.3 字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT | 用户 ID |
| username | VARCHAR(64) | 用户名，唯一 |
| password | VARCHAR(255) | 加密后的密码 |
| nickname | VARCHAR(64) | 用户昵称 |
| avatar_url | VARCHAR(512) | 头像地址 |
| role | VARCHAR(32) | 用户角色 |
| status | TINYINT | 用户状态 |
| is_deleted | TINYINT | 逻辑删除 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

## 5.4 索引说明

| 索引 | 字段 | 说明 |
|---|---|---|
| uk_username | username | 保证用户名唯一 |
| idx_status | status | 支持按用户状态查询 |

## 6. project 项目档案表

## 6.1 表说明

`project` 表用于存储用户创建的项目档案。

项目档案是 AI 项目拷打训练的基础数据来源。

## 6.2 建表 SQL

```sql
CREATE TABLE project (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '项目ID',
    user_id BIGINT NOT NULL COMMENT '所属用户ID',
    name VARCHAR(128) NOT NULL COMMENT '项目名称',
    description TEXT NOT NULL COMMENT '项目描述',
    tech_stack TEXT NOT NULL COMMENT '技术栈',
    role TEXT DEFAULT NULL COMMENT '负责模块',
    highlights TEXT DEFAULT NULL COMMENT '项目亮点',
    difficulties TEXT DEFAULT NULL COMMENT '项目难点',
    status VARCHAR(32) NOT NULL DEFAULT 'NORMAL' COMMENT '状态：NORMAL/DELETED',
    is_deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除：0未删除，1已删除',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted_at DATETIME DEFAULT NULL COMMENT '删除时间',
    KEY idx_user_id (user_id),
    KEY idx_user_deleted_created (user_id, is_deleted, created_at),
    KEY idx_user_name (user_id, name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='项目档案表';
```

## 6.3 字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT | 项目 ID |
| user_id | BIGINT | 所属用户 ID |
| name | VARCHAR(128) | 项目名称 |
| description | TEXT | 项目描述 |
| tech_stack | TEXT | 技术栈 |
| role | TEXT | 负责模块 |
| highlights | TEXT | 项目亮点 |
| difficulties | TEXT | 项目难点 |
| status | VARCHAR(32) | 项目状态 |
| is_deleted | TINYINT | 逻辑删除 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |
| deleted_at | DATETIME | 删除时间 |

## 6.4 索引说明

| 索引 | 字段 | 说明 |
|---|---|---|
| idx_user_id | user_id | 查询某个用户的项目 |
| idx_user_deleted_created | user_id, is_deleted, created_at | 查询用户未删除项目列表并按时间排序 |
| idx_user_name | user_id, name | 支持用户维度下按项目名查询 |

## 6.5 设计说明

1. `description`、`tech_stack`、`role`、`highlights`、`difficulties` 使用 TEXT，方便用户输入较长内容；
2. 使用 `user_id` 做数据隔离；
3. 删除采用逻辑删除；
4. 删除项目后历史训练记录不删除。

## 7. interview_session 训练会话表

## 7.1 表说明

`interview_session` 表用于存储用户发起的一次项目拷打训练。

一次训练会话关联一个项目，包含多轮训练消息，结束后生成训练报告。

## 7.2 建表 SQL

```sql
CREATE TABLE interview_session (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '训练会话ID',
    user_id BIGINT NOT NULL COMMENT '所属用户ID',
    project_id BIGINT NOT NULL COMMENT '项目ID',
    target_role VARCHAR(128) NOT NULL COMMENT '目标岗位',
    difficulty VARCHAR(32) NOT NULL DEFAULT 'NORMAL' COMMENT '难度：EASY/NORMAL/HARD',
    status VARCHAR(32) NOT NULL DEFAULT 'IN_PROGRESS' COMMENT '状态：IN_PROGRESS/FINISHED/FAILED',
    current_round INT NOT NULL DEFAULT 0 COMMENT '当前轮次',
    max_round INT NOT NULL DEFAULT 5 COMMENT '最大轮次',
    started_at DATETIME DEFAULT NULL COMMENT '开始时间',
    ended_at DATETIME DEFAULT NULL COMMENT '结束时间',
    total_score INT DEFAULT NULL COMMENT '总评分',
    is_deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除：0未删除，1已删除',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    KEY idx_user_id (user_id),
    KEY idx_project_id (project_id),
    KEY idx_user_status_created (user_id, status, created_at),
    KEY idx_user_project_created (user_id, project_id, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='训练会话表';
```

## 7.3 字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT | 训练会话 ID |
| user_id | BIGINT | 所属用户 ID |
| project_id | BIGINT | 关联项目 ID |
| target_role | VARCHAR(128) | 目标岗位 |
| difficulty | VARCHAR(32) | 难度 |
| status | VARCHAR(32) | 会话状态 |
| current_round | INT | 当前轮次 |
| max_round | INT | 最大轮次 |
| started_at | DATETIME | 开始时间 |
| ended_at | DATETIME | 结束时间 |
| total_score | INT | 总评分 |
| is_deleted | TINYINT | 逻辑删除 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

## 7.4 索引说明

| 索引 | 字段 | 说明 |
|---|---|---|
| idx_user_id | user_id | 查询用户训练会话 |
| idx_project_id | project_id | 查询项目关联训练会话 |
| idx_user_status_created | user_id, status, created_at | 按用户和状态筛选训练历史 |
| idx_user_project_created | user_id, project_id, created_at | 按项目筛选训练历史 |

## 12. 数据隔离设计

MVP 阶段采用用户级数据隔离。

核心原则：

```text
所有用户业务数据表都必须带 user_id。
```

涉及表：

| 表 | 是否包含 user_id |
|---|---|
| project | 是 |
| interview_session | 是 |
| interview_message | 是 |
| interview_report | 是 |
| ai_call_log | 是 |

查询用户数据时，不能只根据业务 ID 查询，例如：

```sql
SELECT * FROM project WHERE id = #{projectId};
```

必须加上当前用户 ID：

```sql
SELECT * FROM project
WHERE id = #{projectId}
  AND user_id = #{userId}
  AND is_deleted = 0;
```

这样可以避免用户越权访问其他用户的数据。

## 13. 逻辑删除设计

MVP 阶段仅对以下表启用逻辑删除：

| 表 | 字段 |
|---|---|
| user | is_deleted |
| project | is_deleted |
| interview_session | is_deleted |

`interview_message` 和 `interview_report` 暂不单独做逻辑删除。

原因：

1. 消息和报告属于训练会话的从属数据；
2. 训练会话删除后，默认不在页面展示；
3. 保留历史数据便于后续排查问题。

## 14. 设计取舍说明

### 14.1 为什么消息表采用统一消息模型？

训练过程中会产生多种内容：

1. AI 问题；
2. 用户回答；
3. AI 反馈；
4. AI 追问。

如果为每种内容都单独建表，MVP 阶段复杂度较高。

所以第一版采用统一的 `interview_message` 表，通过 `role` 和 `message_type` 区分消息类型。

优点：

1. 对话展示简单；
2. 扩展新消息类型方便；
3. 更接近真实聊天系统设计。

---

### 14.2 为什么报告字段使用 JSON？

训练报告中的优点、薄弱点、建议和问答复盘本质上是结构化列表。

MVP 阶段为了降低表结构复杂度，使用 MySQL JSON 类型保存。

优点：

1. 表结构简单；
2. 适合保存 AI 生成的结构化内容；
3. 后续如果需要统计分析，可以再拆分成报告明细表。

---

### 14.3 为什么没有外键约束？

MVP 阶段暂不使用数据库外键，关联关系由业务代码保证。

原因：

1. Java 后端项目中很多业务系统倾向于不用物理外键；
2. 可以降低后续扩展和数据迁移复杂度；
3. 删除逻辑由业务层控制；
4. 查询仍然通过索引保证性能。

注意：

```text
不使用数据库外键不代表没有关联关系，而是由业务代码维护数据一致性。
```
