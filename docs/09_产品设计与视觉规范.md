# CodeCoach AI 产品设计与视觉规范

## 1. 文档说明

本文档用于定义 CodeCoach AI 的产品定位、完全体信息架构、视觉风格、交互规范和页面设计原则。

当前 CodeCoach AI 已完成 MVP 功能闭环：

用户注册登录 -> 创建项目档案 -> 发起项目拷打训练 -> AI 追问与反馈 -> 结束训练 -> 生成训练报告 -> 查看训练历史。

接下来项目进入产品化打磨阶段。

本阶段目标不是简单“美化页面”，而是将 CodeCoach AI 从一个功能 Demo 打磨成一个具有真实产品质感的 AI 面试训练平台。

---

## 2. 产品定位

CodeCoach AI 的产品定位是：

```text
面向 Java 后端求职者的 AI 项目面试训练工作台。
```

它不是一个普通聊天机器人，也不是一个简单的项目 CRUD 系统，而是围绕“求职训练”设计的 AI 学习产品。

核心价值：

1. 帮助用户沉淀自己的项目经历；
2. 基于项目档案进行 AI 项目拷打训练；
3. 通过多轮追问提升项目表达能力；
4. 通过训练报告发现薄弱点；
5. 后续逐步扩展到八股问答、知识学习、复习计划和能力画像。

---

## 3. 目标用户

### 3.1 核心用户

当前阶段核心用户为：

```text
准备 Java 后端实习、校招、考研复试或转岗面试的学生和初级开发者。
```

典型特征：

1. 有一定 Java 后端基础；
2. 有项目经历，但不一定会表达；
3. 容易在项目面试中被追问卡住；
4. 需要系统化训练项目介绍、技术选型、异常场景和高并发问题；
5. 希望通过 AI 降低自我模拟面试成本。

### 3.2 用户核心痛点

| 痛点 | 说明 |
|---|---|
| 项目会做但不会讲 | 能写代码，但无法清楚描述业务流程和技术决策 |
| 不知道面试官会怎么追问 | 缺少真实项目拷打训练 |
| 八股和项目脱节 | 学了 Redis、MySQL、MQ，但不知道如何结合项目表达 |
| 复盘不系统 | 练完一轮不知道哪里答得不好 |
| 缺少长期训练记录 | 无法观察自己的表达能力是否提升 |

## 6. 设计风格定位

CodeCoach AI 的视觉风格应参考现代海外 SaaS / AI 产品。

参考气质：

```text
Notion 的克制
OpenAI 的干净
Linear 的层级
Vercel 的留白
Raycast 的精致
```

关键词：

```text
简洁
克制
高级
清晰
留白
低饱和
轻量阴影
统一圆角
明确层级
产品化
```

不追求炫酷，而追求：

```text
看起来像一个真实可用的 AI 训练产品。
```

## 7. 不应该做成什么样

CodeCoach AI 不应该做成以下风格：

### 7.1 不要国内后台管理系统风格

避免：

1. 密集表格；
2. 过多边框；
3. 过多彩色按钮；
4. 信息堆叠；
5. 复杂筛选区；
6. 左侧巨型菜单；
7. 没有留白的管理后台感。

### 7.2 不要 AI 生成式低质页面

避免：

1. 大面积蓝紫渐变；
2. 夸张插画；
3. 玻璃拟态泛滥；
4. 过度发光阴影；
5. 没有意义的背景纹理；
6. 过度营销化标题；
7. “看起来很炫但不实用”的页面。

### 7.3 不要普通 Demo 风格

避免：

1. 默认浏览器输入框；
2. 默认 textarea 拖拽；
3. 页面只有一个卡片；
4. 按钮没有主次；
5. 页面之间样式不统一；
6. 没有清晰的信息分区；
7. 没有真实产品的导航和状态反馈。

## 8. 全局布局规范

### 8.1 登录后主布局

MVP 阶段采用：

```text
Top Navigation + Centered Workspace
```

结构：

```text
顶部导航
------------------------------------------------
Logo       项目档案   训练历史   能力报告       用户信息 / 退出

主工作区
------------------------------------------------
最大宽度容器

```

建议：

| 项目 | 建议 |
|---|---|
| 主内容最大宽度 | 1120px - 1200px |
| 页面左右 padding | 24px - 32px |
| 顶部导航高度 | 64px |
| 页面顶部间距 | 32px - 48px |
| 内容卡片间距 | 20px - 24px |

每个登录后页面建议遵循：

```text
PageShell
├── PageHeader
│   ├── Title
│   ├── Description
│   └── Actions
│
└── Content
    ├── Filter / Toolbar，可选
    └── Main Content
```

示例：

```text
项目档案
管理你的项目经历，并用于 AI 面试训练

[搜索项目] [新建项目]

```

## 9. 视觉规范

### 9.1 色彩规范

建议使用低饱和黑白灰体系，少量强调色。

```css
:root {
  --cc-bg: #f7f8fa;
  --cc-bg-subtle: #f9fafb;

  --cc-surface: #ffffff;
  --cc-surface-muted: #f3f4f6;

  --cc-border: #e5e7eb;
  --cc-border-strong: #d1d5db;

  --cc-text-primary: #111827;
  --cc-text-secondary: #4b5563;
  --cc-text-muted: #9ca3af;

  --cc-primary: #111827;
  --cc-primary-hover: #000000;

  --cc-accent: #2563eb;
  --cc-accent-subtle: #eff6ff;

  --cc-danger: #ef4444;
  --cc-success: #16a34a;
  --cc-warning: #d97706;
}
```

使用原则：

1. 文字以黑白灰为主；
2. 主操作按钮可以使用深色或低饱和蓝；
3. 危险操作使用红色，但不要大面积使用；
4. 不使用高饱和大渐变；
5. 背景尽量接近白色或浅灰。

---

### 9.2 字体规范

建议字体栈：

```css
font-family:
  Inter,
  ui-sans-serif,
  system-ui,
  -apple-system,
  BlinkMacSystemFont,
  "Segoe UI",
  "PingFang SC",
  "Microsoft YaHei",
  sans-serif;
```

字号建议：

| 场景 | 字号 | 字重 |
|---|---|---|
| 页面标题 | 28px - 32px | 600 / 700 |
| 分区标题 | 18px - 20px | 600 |
| 卡片标题 | 16px - 18px | 600 |
| 正文 | 14px - 15px | 400 |
| 辅助说明 | 13px - 14px | 400 |
| 标签 / Meta | 12px - 13px | 400 / 500 |

原则：

1. 标题不要过度加粗；
2. 辅助文案使用低对比灰色；
3. 保持行高舒适；
4. 不要在同一页面使用过多字号。

---

### 9.3 圆角规范

```css
--cc-radius-sm: 8px;
--cc-radius-md: 12px;
--cc-radius-lg: 18px;
--cc-radius-xl: 24px;
```

使用建议：

| 组件 | 圆角 |
|---|---|
| 小按钮 / Tag | 8px |
| 输入框 | 10px - 12px |
| 卡片 | 16px - 20px |
| 页面主容器 | 20px - 24px |

---

### 9.4 阴影规范

阴影要轻，不要浓。

```css
--cc-shadow-sm: 0 1px 2px rgba(15, 23, 42, 0.04);
--cc-shadow-card: 0 12px 40px rgba(15, 23, 42, 0.06);
--cc-shadow-popover: 0 20px 60px rgba(15, 23, 42, 0.12);
```

使用原则：

1. 卡片可用轻阴影；
2. 普通输入框不需要阴影；
3. 弹层可以使用稍强阴影；
4. 不要使用彩色阴影。

---

### 9.5 间距规范

建议基础间距：

```css
--cc-space-1: 4px;
--cc-space-2: 8px;
--cc-space-3: 12px;
--cc-space-4: 16px;
--cc-space-5: 20px;
--cc-space-6: 24px;
--cc-space-8: 32px;
--cc-space-10: 40px;
--cc-space-12: 48px;
```

原则：

1. 页面级间距要大；
2. 卡片内 padding 建议 24px - 32px；
3. 表单项间距建议 20px - 24px；
4. 不要让内容贴边；
5. 页面要有呼吸感。

## 10. 组件规范

### 10.1 Button

按钮分为：

| 类型 | 用途 |
|---|---|
| Primary | 主要动作，例如开始训练、保存、提交回答 |
| Secondary | 次要动作，例如编辑、返回 |
| Ghost / Text | 轻量动作，例如取消 |
| Danger | 删除、退出等危险操作 |

原则：

1. 一个区域通常只有一个主按钮；
2. 删除按钮不要和主按钮同等视觉权重；
3. 按钮文案使用动词；
4. loading 状态必须清晰；
5. 禁用状态要明显。

---

### 10.2 Input / TextArea

输入组件是产品质感的重要部分。

统一要求：

1. 输入框圆角统一；
2. 边框低对比；
3. focus 状态清晰但不刺眼；
4. TextArea 禁止用户手动拖拽；
5. TextArea 使用 autoSize；
6. 长文本输入要有合理高度；
7. 表单说明文案要帮助用户理解“为什么填”。

TextArea 规范：

```tsx
<TextArea
  autoSize={{ minRows: 4, maxRows: 8 }}
/>
```

全局 CSS：

```css
textarea,
.ant-input {
  resize: none;
}
```

---

### 10.3 Card

卡片用于承载信息，不是装饰。

卡片规范：

1. 白色背景；
2. 细边框；
3. 大圆角；
4. 轻阴影；
5. 内部间距统一；
6. 卡片之间间距统一；
7. 不要让卡片内容过满。

---

### 10.4 Tag

Tag 用于展示技术栈、状态、难度。

示例：

```text
Spring Boot
React
MySQL
Redis
JWT
NORMAL
FINISHED
```

原则：

1. Tag 颜色低饱和；
2. 不要使用过多颜色；
3. 技术栈 Tag 可以统一灰色；
4. 状态 Tag 可以少量区分颜色。

---

### 10.5 Empty State

空状态不要花哨。

结构：

```text
简洁图标，可选
标题
一句说明
主操作按钮
```

示例：

```text
还没有项目档案
创建你的第一个项目，用于 AI 项目拷打训练。
[新建项目]
```

## 11. 页面级设计规范

### 11.1 项目档案页

页面目标：

```text
让用户管理自己的项目经历，并快速发起训练。
```

页面结构：

```text
PageHeader
├── 标题：项目档案
├── 描述：沉淀你的项目经历，并用于 AI 面试训练
└── 新建项目按钮

Toolbar
├── 搜索框
└── 筛选项，后续扩展

Project List
└── Project Card
```

项目卡片建议字段：

1. 项目名称；
2. 一句话描述；
3. 技术栈 Tag；
4. 负责模块摘要；
5. 最近更新时间；
6. 后续可增加训练次数 / 最近分数；
7. 主操作：开始训练；
8. 次操作：编辑、删除。

设计目标：

1. 像“项目资产库”，不是 CRUD 列表；
2. 强调“开始训练”；
3. 技术栈信息要清晰；
4. 项目描述要有摘要感。

---

### 11.2 项目创建 / 编辑页

页面目标：

```text
帮助用户构建高质量项目档案，让 AI 后续能生成更精准的追问。
```

推荐布局：

```text
页面标题区
表单主体卡片
├── 基本信息
│   ├── 项目名称
│   └── 项目描述
│
├── 技术信息
│   ├── 技术栈
│   └── 负责模块
│
└── 面试素材
    ├── 项目亮点
    └── 项目难点
```

可以增加轻量说明：

```text
项目亮点将用于生成“为什么这么设计”的追问。
项目难点将用于生成“异常场景、性能、并发和一致性”的追问。
```

设计目标：

1. 不要像后台表单；
2. 要像“构建面试素材”；
3. 分区清晰；
4. 输入体验舒适；
5. TextArea 不可拖拽；
6. 保存按钮固定在表单底部或右下区域。

---

### 11.3 训练页 Interview Studio

页面目标：

```text
让用户进入沉浸式 AI 项目面试训练。
```

推荐结构：

```text
顶部训练信息栏
├── 项目名称
├── 目标岗位
├── 难度
└── 当前轮次 / 最大轮次

中间消息流
├── AI 问题
├── 用户回答
├── AI 反馈
└── AI 追问

底部回答输入区
├── TextArea composer
└── 提交按钮
```

设计重点：

1. 训练页应该像 AI 产品，而不是普通表单；
2. AI 消息和用户消息要有清晰区分；
3. 反馈消息可以有轻微特殊样式；
4. 输入区要像 ChatGPT / Claude 的 composer；
5. 提交中必须显示“AI 正在思考”；
6. 禁止重复提交；
7. 最后一轮完成后自动进入报告。

消息类型展示建议：

| messageType | 展示方式 |
|---|---|
| AI_QUESTION | 面试官问题 |
| USER_ANSWER | 我的回答 |
| AI_FEEDBACK | 本轮反馈 |
| AI_FOLLOW_UP | 追问 |
| SYSTEM_NOTICE | 系统提示 |

---

### 11.4 报告页

页面目标：

```text
让用户清楚知道自己哪里答得好、哪里不足、下一步怎么提升。
```

推荐结构：

```text
报告头部
├── 项目名称
├── 目标岗位
├── 难度
├── 训练时间
└── 总分

总体评价 Summary

能力反馈区
├── 优点 Strengths
├── 薄弱点 Weaknesses
└── 改进建议 Suggestions

问答复盘 QA Review
└── 每轮问题 / 回答 / 反馈
```

设计目标：

1. 像正式分析报告；
2. 总分要醒目但不夸张；
3. 优点、薄弱点、建议要有清晰区分；
4. 问答复盘要可读；
5. 后续可扩展能力雷达图、薄弱点标签、复习计划。

---

### 11.5 训练历史页

页面目标：

```text
让用户查看过往训练记录，并继续未完成训练或查看报告。
```

结构建议：

```text
PageHeader
├── 标题：训练历史
└── 描述：查看你的项目拷打记录和训练报告

Filter
├── 状态筛选
├── 项目筛选，后续
└── 时间筛选，后续

History List
└── Record Card / Table
```

每条记录展示：

1. 项目名称；
2. 目标岗位；
3. 难度；
4. 状态；
5. 当前轮次；
6. 总分；
7. 创建时间；
8. 结束时间；
9. 操作：继续训练 / 查看报告。

设计目标：

1. 不要过度表格化；
2. 记录状态要清楚；
3. 用户能快速知道下一步能做什么。

---

### 11.6 登录 / 注册页

页面目标：

```text
让用户快速进入产品，不做复杂营销。
```

设计建议：

1. 居中卡片；
2. 背景简洁；
3. 表单宽度 360px - 420px；
4. Logo 和产品名克制展示；
5. 不要复杂插画；
6. 不要大面积渐变；
7. 注册页和登录页保持统一。

## 12. 交互规范

### 12.1 Loading

所有异步操作必须有 loading 状态：

| 操作 | Loading 表现 |
|---|---|
| 页面加载 | Skeleton / Spin |
| 表单提交 | 按钮 loading |
| AI 生成 | 明确提示“AI 正在思考” |
| 删除 | 按钮 loading 或确认框 loading |
| 报告生成 | 展示“正在生成报告” |

### 12.2 错误提示

错误提示使用统一 message / notification。

原则：

1. 错误文案要清楚；
2. 不直接展示技术异常；
3. AI 调用失败提示“AI 调用失败，请稍后重试”；
4. Token 失效跳转登录；
5. 表单错误优先在字段下方展示。

### 12.3 空状态

所有列表页都要处理空状态：

1. 项目为空；
2. 训练历史为空；
3. 报告数组为空；
4. 问答复盘为空。

### 12.4 重复提交

所有写操作必须防重复提交：

1. 创建项目；
2. 修改项目；
3. 删除项目；
4. 提交回答；
5. 结束训练；
6. 再来一次。

AI 相关操作尤其需要防重复。

## 13. AI 产品交互规范

AI 功能不是普通接口，需要额外设计。

### 13.1 AI 等待状态

提交回答后，需要提示：

```text
AI 正在分析你的回答，这可能需要几秒钟。
```

要求：

1. 禁用提交按钮；
2. 禁用或锁定输入框；
3. 不要让用户误以为页面卡死；
4. 请求超时时间要比普通接口更长；
5. 后端必须防重复提交。

### 13.2 AI 失败状态

如果 AI 调用失败：

```text
AI 调用失败，请稍后重试。
```

不要 fallback 到 Mock 内容。

原因：

1. 线上 Mock 会让用户误以为产品质量很差；
2. 真实失败应该明确告诉用户；
3. Mock 只用于开发和测试环境。

### 13.3 AI 输出结构化

真实模型输出应尽量使用 JSON 结构：

1. 反馈和追问；
2. 训练报告；
3. 薄弱点；
4. 建议；
5. 问答复盘。

后端负责解析和兜底。

## 17. 工程约束

UI 改造过程中必须遵守：

1. 不修改后端接口；
2. 不破坏路由；
3. 不破坏登录态；
4. 不删除已有业务功能；
5. 不引入大型 UI 框架替代 Ant Design；
6. 不引入复杂动画库，除非确有必要；
7. 不使用大量 inline style；
8. 不制造重复组件；
9. 不把所有样式写在页面里；
10. 不为了视觉效果牺牲可维护性。
