# CodeCoach AI 技术选型与项目结构

## 1. 文档说明

本文档用于描述 CodeCoach AI MVP 版本的技术选型、项目结构、开发环境和工程组织方式。

MVP 版本核心目标是完成“项目拷打训练闭环”：

用户注册登录 -> 创建项目档案 -> 发起项目拷打训练 -> AI 生成面试追问 -> 用户回答 -> AI 反馈追问 -> 结束训练 -> 生成训练报告。

---

## 2. 总体架构

MVP 阶段采用前后端分离的单体应用架构。

```text
React 前端
    |
    | HTTP / JSON
    v
Spring Boot 后端
    |
    | JDBC / MyBatis-Plus
    v
MySQL
    |
    | RedisTemplate，可选
    v
Redis
    |
    | HTTP API
    v
外部大模型服务
```

MVP 阶段暂不引入微服务、消息队列、复杂网关、注册中心和分布式事务。

---

## 3. 架构选择说明

## 3.1 为什么选择单体架构？

MVP 阶段选择单体架构，原因如下：

1. 项目第一版业务范围较小；
2. 单体架构开发、调试、部署成本更低；
3. 更适合个人开发和快速上线；
4. 可以通过模块化包结构保持代码清晰；
5. 后续如果业务复杂，可以再拆分为微服务。

## 3.2 为什么选择前后端分离？

原因如下：

1. 更接近真实企业开发流程；
2. 前端和后端可以独立开发；
3. API 文档可以作为协作边界；
4. 后续部署时前端可由 Nginx 托管，后端独立运行；
5. 方便后续接入移动端或其他客户端。

## 4.2 Java 版本选择

后端使用 Java 21。

选择原因：

1. Java 21 是长期支持版本；
2. Spring Boot 3.x 支持良好；
3. 语法和性能相比 Java 8 更现代；
4. 符合当前较新的 Java 后端开发趋势。

---

## 4.3 Spring Boot 选择

使用 Spring Boot 3.x。

主要能力：

1. 快速搭建 Web 应用；
2. 集成 Spring MVC；
3. 集成 Spring Security；
4. 管理配置文件；
5. 方便后续接入 Redis、MySQL、AI 服务等。

---

## 4.4 持久层选择

使用 MyBatis-Plus。

原因：

1. 适合 Java 后端项目；
2. 简化基础 CRUD；
3. 保留手写 SQL 能力；
4. 适合学习 MyBatis 和数据库设计；
5. 面试中容易讲清楚。

---

## 4.5 认证方案选择

MVP 阶段使用 Spring Security + JWT。

基本流程：

```text
用户登录
    |
    v
后端校验用户名密码
    |
    v
生成 JWT
    |
    v
前端保存 Token
    |
    v
后续请求携带 Authorization 请求头
    |
    v
后端解析 Token，获取当前用户 ID
```

请求头格式：

```http
Authorization: Bearer <token>
```

## 4.6 AI 模型接入策略

MVP 阶段采用外部大模型 API，不进行本地模型部署。

当前优先接入 Xiaomi MiMo，原因是：

1. 已获得较大 Token 额度，适合 MVP 阶段开发和测试；
2. 支持 API Key 调用；
3. 可用于生成项目追问、回答反馈和训练报告；
4. 成本较低，适合个人项目上线演示。

为了避免项目绑定某一家模型厂商，后端需要设计统一的 AI 服务接口。

第一阶段使用 Mock AI 实现，保证无 API Key 时也能完成前后端联调。

第二阶段接入 Xiaomi MiMo。

第三阶段支持 OpenAI-Compatible 模型切换，例如 DeepSeek、通义千问、OpenRouter 等。

建议封装统一接口：

```java
public interface AiChatService {

    String generateQuestion(Project project, InterviewSession session);

    AiAnswerResponse generateFeedbackAndNextQuestion(InterviewContext context);

    InterviewReportContent generateReport(InterviewContext context);
}
```

业务代码不要直接依赖某一家模型厂商的 SDK。

---

## 4.7 Redis 使用策略

MVP 阶段 Redis 可先接入但不强依赖。

可用于：

1. 登录 Token 黑名单，后续扩展；
2. 接口限流，后续扩展；
3. AI 调用防重复提交，后续扩展；
4. 缓存用户信息，后续扩展。

第一版如果时间紧，可以先只启动 Redis，不写复杂业务逻辑。

## 5.2 为什么选择 React + TypeScript？

原因：

1. React 生态成熟；
2. TypeScript 能提升项目可维护性；
3. 更接近真实前端工程；
4. 适合与 AI 编程工具协作；
5. 方便后续扩展复杂页面。

---

## 5.3 为什么选择 Ant Design？

原因：

1. 表单、表格、弹窗、消息提示组件完善；
2. 适合后台管理和工具类平台；
3. 开发效率高；
4. MVP 阶段可以快速搭建可用页面。

---

## 5.4 前端状态管理

MVP 阶段使用 Zustand。

主要存储：

1. 当前登录用户信息；
2. Token；
3. 全局 loading 状态，可选。

复杂业务状态尽量放在页面组件内部，不提前过度设计。

## 16. 技术风险与应对

## 16.1 AI 输出不稳定

风险：

```text
大模型输出格式可能不稳定，影响前端展示。
```

应对：

1. Prompt 中要求输出固定 JSON 结构；
2. 后端对 AI 输出进行 JSON 解析和兜底；
3. 解析失败时返回默认反馈；
4. 保存原始 AI 输出，方便排查。

---

## 16.2 AI 接口调用失败

风险：

```text
外部模型 API 可能超时、限流或失败。
```

应对：

1. 设置请求超时时间；
2. 捕获异常并返回友好提示；
3. 记录 AI 调用日志；
4. 后续支持重试机制。

---

## 16.3 用户数据越权

风险：

```text
用户可能通过修改 ID 访问他人的项目、会话或报告。
```

应对：

1. 所有业务查询都带当前 userId；
2. userId 从 Token 解析，不由前端传入；
3. Service 层统一做资源归属校验。

---

## 16.4 前后端接口不一致

风险：

```text
前端字段名和后端 DTO 字段名不一致，导致联调成本增加。
```

应对：

1. 以 `05_API接口设计.md` 为准；
2. 后端优先生成 OpenAPI 文档；
3. 前端按接口文档定义 TypeScript 类型；
4. 接口变更必须同步修改文档。

---
