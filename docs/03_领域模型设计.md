# CodeCoach AI 领域模型设计

## 1. 文档说明

本文档用于描述 CodeCoach AI MVP 版本的核心领域模型、模型关系、字段设计和状态流转。

领域模型是业务系统的核心抽象，后续数据库设计、接口设计、后端代码结构都会基于本文档展开。

MVP 版本核心业务闭环为：

用户创建项目档案 -> 发起项目拷打训练 -> AI 生成问题 -> 用户回答 -> AI 反馈和追问 -> 用户结束训练 -> 系统生成训练报告。

---

## 2. 核心领域对象总览

MVP 版本包含以下核心领域对象：

| 领域对象 | 说明 |
|---|---|
| User | 用户 |
| Project | 项目档案 |
| InterviewSession | 训练会话 |
| InterviewMessage | 训练消息 |
| InterviewReport | 训练报告 |
| AiCallLog | AI 调用日志 |

对象关系概览：

```text
User 1 --- n Project

User 1 --- n InterviewSession

Project 1 --- n InterviewSession

InterviewSession 1 --- n InterviewMessage

InterviewSession 1 --- 1 InterviewReport

User 1 --- n AiCallLog
```

---

## 3. User 用户模型

## 3.1 模型说明

User 表示系统中的注册用户。

MVP 阶段用户主要用于登录认证和数据隔离。每个用户只能访问自己创建的项目、训练会话和训练报告。

## 3.2 核心字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | Long | 用户 ID |
| username | String | 用户名，唯一 |
| password | String | 加密后的密码 |
| nickname | String | 用户昵称 |
| avatarUrl | String | 用户头像地址，MVP 可为空 |
| role | String | 用户角色 |
| status | Integer | 用户状态 |
| createdAt | LocalDateTime | 创建时间 |
| updatedAt | LocalDateTime | 更新时间 |

## 3.3 角色枚举

| 值 | 说明 |
|---|---|
| USER | 普通用户 |
| ADMIN | 管理员，MVP 暂不实现具体能力 |

## 3.4 状态枚举

| 值 | 说明 |
|---|---|
| 1 | 正常 |
| 0 | 禁用 |

## 3.5 业务规则

1. 用户名必须唯一；
2. 密码不能明文存储；
3. 用户登录成功后返回 JWT；
4. 用户只能访问自己的业务数据；
5. MVP 阶段不做复杂角色权限体系。

---

## 4. Project 项目档案模型

## 4.1 模型说明

Project 表示用户创建的项目档案。

项目档案是项目拷打训练的基础数据来源，AI 会根据项目名称、项目描述、技术栈、负责模块、项目亮点和项目难点生成面试追问。

## 4.2 核心字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | Long | 项目 ID |
| userId | Long | 所属用户 ID |
| name | String | 项目名称 |
| description | String | 项目描述 |
| techStack | String | 技术栈 |
| role | String | 用户负责模块 |
| highlights | String | 项目亮点 |
| difficulties | String | 项目难点 |
| status | String | 项目状态 |
| createdAt | LocalDateTime | 创建时间 |
| updatedAt | LocalDateTime | 更新时间 |
| deletedAt | LocalDateTime | 删除时间，逻辑删除时使用 |

## 4.3 状态枚举

| 值 | 说明 |
|---|---|
| NORMAL | 正常 |
| DELETED | 已删除 |

## 4.4 业务规则

1. 项目必须属于某个用户；
2. 项目名称、项目描述、技术栈不能为空；
3. 用户只能查看、修改、删除自己的项目；
4. 删除项目采用逻辑删除；
5. 删除项目后，历史训练记录仍然保留；
6. 项目内容越完整，AI 生成的追问质量越高。

---

## 5. InterviewSession 训练会话模型

## 5.1 模型说明

InterviewSession 表示一次项目拷打训练。

用户基于某个项目创建训练会话后，系统会围绕该项目进行多轮 AI 面试追问。一次训练会话包含多条训练消息，结束后生成一份训练报告。

## 5.2 核心字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | Long | 训练会话 ID |
| userId | Long | 所属用户 ID |
| projectId | Long | 关联项目 ID |
| targetRole | String | 目标岗位，例如 Java 后端实习 |
| difficulty | String | 训练难度 |
| status | String | 会话状态 |
| currentRound | Integer | 当前轮次 |
| maxRound | Integer | 最大轮次 |
| startedAt | LocalDateTime | 开始时间 |
| endedAt | LocalDateTime | 结束时间 |
| createdAt | LocalDateTime | 创建时间 |
| updatedAt | LocalDateTime | 更新时间 |

## 5.3 难度枚举

| 值 | 说明 |
|---|---|
| EASY | 简单 |
| NORMAL | 普通 |
| HARD | 困难 |

## 5.4 状态枚举

| 值 | 说明 |
|---|---|
| IN_PROGRESS | 训练中 |
| FINISHED | 已完成 |
| FAILED | 生成失败 |

## 5.5 状态流转

```text
创建训练会话
    |
    v
IN_PROGRESS
    |
    |-- 用户正常结束训练
    |       v
    |   FINISHED
    |
    |-- AI 调用失败 / 系统异常
            v
        FAILED
```

## 5.6 业务规则

1. 训练会话必须属于某个用户；
2. 训练会话必须关联某个项目；
3. 只能基于自己的项目创建训练会话；
4. 会话创建后状态为 IN_PROGRESS；
5. 会话结束后状态为 FINISHED；
6. FINISHED 状态下不能继续提交回答；
7. FAILED 状态下可在后续版本支持重试；
8. MVP 阶段默认最大轮次为 5 轮。

---

## 6. InterviewMessage 训练消息模型

## 6.1 模型说明

InterviewMessage 表示训练过程中的一条消息。

一次训练会话中会产生多条消息，包括 AI 提问、用户回答、AI 反馈和 AI 追问。

MVP 阶段为了降低复杂度，可以把问题、回答、反馈、追问都统一抽象为消息。

## 6.2 核心字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | Long | 消息 ID |
| sessionId | Long | 所属训练会话 ID |
| userId | Long | 所属用户 ID |
| role | String | 消息角色 |
| messageType | String | 消息类型 |
| content | Text | 消息内容 |
| roundNo | Integer | 所属轮次 |
| createdAt | LocalDateTime | 创建时间 |

## 6.3 消息角色枚举

| 值 | 说明 |
|---|---|
| USER | 用户 |
| ASSISTANT | AI 助手 |
| SYSTEM | 系统 |

## 6.4 消息类型枚举

| 值 | 说明 |
|---|---|
| AI_QUESTION | AI 面试问题 |
| USER_ANSWER | 用户回答 |
| AI_FEEDBACK | AI 对回答的反馈 |
| AI_FOLLOW_UP | AI 继续追问 |
| SYSTEM_NOTICE | 系统提示 |

## 6.5 业务规则

1. 消息必须属于某个训练会话；
2. 消息必须记录所属用户，便于数据隔离和查询；
3. 用户提交回答时生成 USER_ANSWER 消息；
4. AI 生成问题时生成 AI_QUESTION 消息；
5. AI 对回答进行评价时生成 AI_FEEDBACK 消息；
6. AI 继续追问时生成 AI_FOLLOW_UP 消息；
7. 消息按创建时间正序展示；
8. 已结束会话不能继续新增用户回答消息。

---

## 7. InterviewReport 训练报告模型

## 7.1 模型说明

InterviewReport 表示一次训练结束后生成的反馈报告。

报告基于训练会话中的项目档案和多轮问答生成，主要用于帮助用户发现项目表达中的优点、薄弱点和改进方向。

## 7.2 核心字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | Long | 报告 ID |
| sessionId | Long | 训练会话 ID |
| userId | Long | 所属用户 ID |
| projectId | Long | 关联项目 ID |
| totalScore | Integer | 总评分，0-100 |
| summary | Text | 总体评价 |
| strengths | Text / JSON | 优点列表 |
| weaknesses | Text / JSON | 薄弱点列表 |
| suggestions | Text / JSON | 改进建议列表 |
| qaReview | Text / JSON | 问答复盘 |
| createdAt | LocalDateTime | 创建时间 |

## 7.3 业务规则

1. 一次训练会话最多生成一份训练报告；
2. 只有 FINISHED 状态的会话才能生成报告；
3. 报告必须属于当前用户；
4. 用户只能查看自己的报告；
5. MVP 阶段报告内容可使用 JSON 字符串保存；
6. 总评分范围为 0-100。

---

## 8. AiCallLog AI 调用日志模型

## 8.1 模型说明

AiCallLog 用于记录每次调用大模型接口的情况。

该模型用于后续排查问题、统计模型调用次数、分析响应耗时和成本。

MVP 阶段可以先设计表结构，核心链路跑通后再逐步完善日志记录。

## 8.2 核心字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | Long | 日志 ID |
| userId | Long | 用户 ID |
| sessionId | Long | 训练会话 ID，可为空 |
| provider | String | 模型服务商 |
| modelName | String | 模型名称 |
| requestType | String | 调用类型 |
| promptTokens | Integer | 输入 Token 数 |
| completionTokens | Integer | 输出 Token 数 |
| totalTokens | Integer | 总 Token 数 |
| latencyMs | Long | 调用耗时，毫秒 |
| success | Boolean | 是否成功 |
| errorMessage | Text | 错误信息 |
| createdAt | LocalDateTime | 创建时间 |

## 8.3 调用类型枚举

| 值 | 说明 |
|---|---|
| GENERATE_QUESTION | 生成面试问题 |
| GENERATE_FEEDBACK | 生成回答反馈 |
| GENERATE_REPORT | 生成训练报告 |

## 8.4 业务规则

1. 每次调用 AI 接口都应尽量记录日志；
2. 调用成功时记录模型、耗时和 token 用量；
3. 调用失败时记录错误信息；
4. 日志不影响主流程，日志保存失败不应导致用户操作失败。

## 9. 领域模型关系说明

## 9.1 User 与 Project

```text
一个用户可以创建多个项目。
一个项目只能属于一个用户。
```

关系：

```text
User 1 --- n Project
```

业务约束：

```text
查询、修改、删除项目时，必须同时校验 projectId 和 userId。
```

---

## 9.2 User 与 InterviewSession

```text
一个用户可以发起多次训练会话。
一次训练会话只能属于一个用户。
```

关系：

```text
User 1 --- n InterviewSession
```

业务约束：

```text
查询训练会话时，必须校验 sessionId 和 userId。
```

---

## 9.3 Project 与 InterviewSession

```text
一个项目可以发起多次训练会话。
一次训练会话只能基于一个项目。
```

关系：

```text
Project 1 --- n InterviewSession
```

业务约束：

```text
创建训练会话时，需要校验项目是否属于当前用户。
```

---

## 9.4 InterviewSession 与 InterviewMessage

```text
一次训练会话包含多条训练消息。
一条训练消息只能属于一次训练会话。
```

关系：

```text
InterviewSession 1 --- n InterviewMessage
```

业务约束：

```text
展示训练对话时，按 sessionId 查询消息列表，并按 createdAt 正序排序。
```

---

## 9.5 InterviewSession 与 InterviewReport

```text
一次训练会话最多对应一份训练报告。
一份训练报告只能属于一次训练会话。
```

关系：

```text
InterviewSession 1 --- 0/1 InterviewReport
```

业务约束：

```text
结束训练时，如果报告已存在，不应重复生成。
```

---

## 10. 核心业务不变量

系统中必须长期保持以下业务规则成立：

1. 用户只能访问自己的数据；
2. 项目必须属于某个用户；
3. 训练会话必须属于某个用户；
4. 训练会话必须关联某个项目；
5. 训练消息必须属于某个训练会话；
6. 训练报告必须属于某个训练会话；
7. 已结束训练不能继续提交回答；
8. 已删除项目不应出现在项目列表中；
9. 删除项目不应删除历史训练报告；
10. AI 调用失败时不能破坏已有业务数据。
