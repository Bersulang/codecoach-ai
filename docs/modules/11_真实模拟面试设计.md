# 真实模拟面试设计

## 1. 文档说明

本文档用于设计 CodeCoach AI 的真实模拟面试模块。

当前 CodeCoach AI 已经完成：

```text
项目拷打训练
八股问答训练
知识学习模块
成长洞察 / 能力画像
RAG 知识文章增强
RAG 项目档案增强
用户文档 RAG
简历驱动训练
面试复盘 Agent
产品内 AI 训练向导
token 级流式输出
评分 Rubric 与低质量回答判定
```

现在需要新增一个更贴近真实面试场景的功能：

```text
真实模拟面试
```

它不是单点专项训练，而是一场完整的、多阶段的、限时的、基于用户简历和项目材料的模拟面试。

---

## 2. 模块定位

真实模拟面试的定位是：

```text
面向 Java 后端求职场景的综合模拟面试模块。
```

它的目标是模拟真实一面 / 技术面中的完整流程：

```text
开场自我介绍
简历项目追问
技术八股提问
项目深挖
场景题 / 设计题
反问与总结
最终综合报告
```

它不是：

```text
普通聊天
单个八股题训练
单个项目拷打
简历分析工具
随机刷题工具
面试题库页面
```

它应该让用户体验到：

```text
我正在经历一场真实技术面试。
```

---

## 3. 为什么需要真实模拟面试

当前系统已有多个训练模块：

```text
八股问答
项目拷打
简历训练
复盘 Agent
能力画像
```

但这些模块更多是专项训练。

真实面试并不会只问某一类问题。它通常会混合：

```text
自我介绍
简历项目
项目细节
技术原理
场景设计
代码习惯
工程权衡
反问
```

用户需要一种能力：

```text
把分散训练能力整合到一场完整面试中。
```

真实模拟面试可以解决这些问题：

```text
用户不知道自己综合面试表现如何
用户在专项训练中表现还可以，但真实面试容易断节奏
用户不会从简历切换到技术细节
用户不知道回答节奏是否合适
用户缺少真实面试压力感
用户需要面试前最后一次综合演练
```

---

## 4. 产品核心目标

真实模拟面试第一版需要实现：

```text
一场完整模拟面试
多阶段问题编排
限时体验
token 级流式追问
基于简历 / 项目 / 文档 / 知识库的 RAG
面试结束后生成综合报告
报告沉淀能力画像
```

最终用户应该能完成：

```text
选择模拟面试类型
选择目标岗位和难度
选择简历 / 项目材料
开始模拟面试
按真实面试节奏回答问题
面试结束后查看综合评价
获得下一步训练建议
```

---

## 5. 模拟面试与现有训练的区别

## 5.1 八股问答

八股问答关注：

```text
某个知识点是否会答
```

真实模拟面试关注：

```text
用户能不能在真实面试语境中完整表达
```

---

## 5.2 项目拷打

项目拷打关注：

```text
某个项目是否讲得清楚
```

真实模拟面试关注：

```text
用户能否把简历项目、技术原理、工程权衡和追问应对串起来
```

---

## 5.3 简历训练

简历训练关注：

```text
简历上哪些点容易被追问
```

真实模拟面试关注：

```text
面试官如何基于简历发起一整场综合面试
```

---

## 5.4 复盘 Agent

复盘 Agent 关注：

```text
最近训练表现有什么模式
```

真实模拟面试关注：

```text
单场完整面试表现如何
```

---

## 6. 模拟面试类型

第一版建议支持三种类型。

---

## 6.1 综合技术一面

适合：

```text
Java 后端实习 / 初级后端面试
```

问题组成：

```text
自我介绍
简历项目
Java / Spring / MySQL / Redis 等基础技术
项目细节追问
场景题
反问
```

这是第一版最推荐的默认模式。

---

## 6.2 简历项目深挖面

适合：

```text
针对简历中某个项目进行深挖
```

问题组成：

```text
项目背景
个人职责
技术选型
核心难点
异常场景
性能优化
项目反思
```

该模式可以复用项目拷打和简历训练能力。

---

## 6.3 八股综合面

适合：

```text
面试前快速检测基础知识
```

问题组成：

```text
Java 基础
JVM
JUC
MySQL
Redis
Spring
计算机网络
操作系统
```

该模式比专项八股训练更混合。

---

## 7. 面试阶段设计

真实模拟面试不应该随机连续提问，而应有阶段。

建议第一版阶段：

```text
OPENING
RESUME_PROJECT
TECHNICAL_FUNDAMENTAL
PROJECT_DEEP_DIVE
SCENARIO_DESIGN
WRAP_UP
```

---

## 8. OPENING：开场阶段

目标：

```text
让用户进入真实面试状态。
```

问题示例：

```text
请先做一个 1 分钟左右的自我介绍，重点说一下你和 Java 后端相关的项目经历。
```

评估重点：

```text
表达结构
目标岗位匹配度
项目引入能力
简洁性
```

---

## 9. RESUME_PROJECT：简历项目阶段

目标：

```text
根据用户简历和项目材料追问项目经历。
```

问题示例：

```text
你简历里提到 CodeCoach AI 这个项目，我想先了解一下这个项目主要解决什么问题，你在里面负责了哪些部分？
```

评估重点：

```text
项目背景
个人贡献
真实性
项目边界
表达可信度
```

---

## 10. TECHNICAL_FUNDAMENTAL：基础技术阶段

目标：

```text
考察 Java 后端基础知识。
```

问题示例：

```text
你项目中用了 Redis，那你能说一下缓存击穿和缓存穿透的区别吗？
```

评估重点：

```text
概念准确性
原理理解
场景适配
表达清晰度
```

---

## 11. PROJECT_DEEP_DIVE：项目深挖阶段

目标：

```text
深入追问项目中的核心技术细节。
```

问题示例：

```text
你提到系统接入了 RAG，那么文档从上传到被检索出来的完整链路是怎样的？
```

评估重点：

```text
技术实现细节
工程权衡
异常处理
降级方案
可观测性
```

---

## 12. SCENARIO_DESIGN：场景设计阶段

目标：

```text
考察用户解决开放工程问题的能力。
```

问题示例：

```text
如果你们的用户文档越来越多，RAG 索引耗时变长，你会如何优化上传后的解析和索引流程？
```

评估重点：

```text
架构思考
扩展性
异步化
性能意识
稳定性
```

---

## 13. WRAP_UP：收尾阶段

目标：

```text
模拟真实面试结束前的总结和反问。
```

问题示例：

```text
你还有什么想补充的，或者你觉得这个项目最能体现你能力的地方是什么？
```

评估重点：

```text
总结能力
自我认知
亮点提炼
表达收束
```

## 14. 面试配置

用户开始模拟面试前，需要配置：

```text
面试类型
目标岗位
难度
面试时长
是否使用简历
是否使用项目文档
选择关联项目
选择关联简历
```

第一版配置可以简化为：

```text
面试类型
目标岗位
难度
选择项目，可选
选择简历，可选
面试轮数
```

---

## 15. 难度设计

建议支持：

```text
EASY
NORMAL
HARD
```

含义：

```text
EASY：偏基础，适合首次模拟
NORMAL：接近常规实习 / 初级后端面试
HARD：追问更深，更强调项目真实性和工程细节
```

---

## 16. 面试轮数与时间

第一版可以用轮数控制，不强依赖真实倒计时。

建议：

```text
快速模拟：5 轮
标准模拟：8 轮
深度模拟：12 轮
```

后续再加入真实倒计时：

```text
15 分钟
30 分钟
45 分钟
```

---

## 17. 问题生成策略

真实模拟面试的问题不应该完全随机。

生成策略应考虑：

```text
面试阶段
用户简历风险点
用户项目档案
用户上传文档
能力画像薄弱点
历史训练表现
当前回答内容
```

问题编排示例：

```text
第 1 轮：自我介绍
第 2 轮：简历项目概述
第 3 轮：项目职责追问
第 4 轮：项目技术细节
第 5 轮：相关八股知识
第 6 轮：异常场景
第 7 轮：开放设计题
第 8 轮：总结反问
```

---

## 18. RAG 使用策略

真实模拟面试应综合使用多种 RAG 来源。

可检索：

```text
PROJECT 项目档案
USER_UPLOAD 简历和项目文档
KNOWLEDGE_ARTICLE 知识文章
resume_analysis 简历风险点
历史训练报告，后续
```

不同阶段使用不同来源：

| 阶段 | 主要 RAG 来源 |
|---|---|
| OPENING | resume_analysis / project |
| RESUME_PROJECT | USER_UPLOAD / PROJECT |
| TECHNICAL_FUNDAMENTAL | KNOWLEDGE_ARTICLE |
| PROJECT_DEEP_DIVE | PROJECT / USER_UPLOAD |
| SCENARIO_DESIGN | PROJECT / KNOWLEDGE_ARTICLE |
| WRAP_UP | 全局摘要 |

---

## 19. AI 面试官角色

AI 在模拟面试中应扮演：

```text
严谨但不刻薄的 Java 后端面试官
```

风格：

```text
追问具体
关注真实性
不轻易放过模糊回答
不过度打断
不羞辱用户
不直接给答案
每轮只问一个主问题
必要时指出回答不完整
```

---

## 20. 回答反馈策略

真实模拟面试和专项训练不同。

专项训练可以每轮给详细反馈。

真实模拟面试第一版建议：

```text
面试过程中只给简短追问或过渡
不立即展示完整参考答案
面试结束后统一生成综合报告
```

原因：

```text
真实面试不会每答一题就给标准答案
这样更接近真实体验
```

但为了产品体验，可以在每轮后给轻量提示：

```text
好的，我继续追问一下……
这个点你提到了方案，但我想进一步确认……
```

不要每轮给大段教学反馈。

---

## 21. 面试过程中的 AI 输出

每轮 AI 输出应包括：

```text
面试官回应
下一问题
当前阶段
是否进入下一阶段
```

前端展示时更像真实对话：

```text
面试官：你刚才提到了 Redis 缓存一致性，我继续追问一下，如果缓存更新成功但数据库更新失败，你怎么处理？
```

---

## 22. 真实感设计

真实模拟面试需要一些真实感。

建议加入：

```text
阶段标签
当前轮次
面试官语气
简短追问
不展示参考答案
结束后统一报告
```

避免：

```text
每轮都像教学题解析
每轮都给标准答案
问题之间没有上下文
完全随机八股题
```

---

## 23. 结束条件

面试结束条件：

```text
达到最大轮数
用户主动结束
AI 判断已完成主要阶段
异常中断后手动结束
```

第一版以最大轮数为主。

建议维度：

```text
综合表达

```

## 26. 评分 Rubric

评分需要复用现有评分体系原则：

```text
低质量回答不能虚高
不知道 / 不会要低分
样本不足要提示
必须有扣分原因
必须给下一步建议
```

真实模拟面试还应关注：

```text
用户是否能连续回答
是否能承接上下文
是否在追问中保持一致
是否能从项目讲到技术原理
是否能承认不确定而不是胡编
```

---

## 27. 阶段表现

报告中应展示每个阶段表现。

示例：

```json
[
  {
    "stage": "RESUME_PROJECT",
    "score": 68,
    "comment": "能讲清项目背景，但个人贡献和关键技术决策表达不够具体。"
  },
  {
    "stage": "PROJECT_DEEP_DIVE",
    "score": 52,
    "comment": "面对 RAG 链路追问时，能说出模块名，但缺少异常处理和数据一致性细节。"
  }
]
```

---

## 28. 高风险回答

报告应指出高风险回答。

示例：

```text
你多次使用“我们做了性能优化”这类表述，但没有说明具体指标、瓶颈和优化手段。
```

## 29. 面试节奏评估

真实模拟面试可以评估：

```text

是否经常答非所问
是否能在追问中补充细节
是否会主动总结
```

## 41. 前端交互原则

模拟面试过程中：

```text
不要每轮展示参考答案
不要把页面做成练习题解析
不要频繁弹出报告
不要让用户感觉是普通聊天
```

应该强调：

```text
面试官正在追问
当前处于哪个阶段
用户需要像真实面试一样回答
结束后统一复盘
```

---

## 42. token 级流式输出

模拟面试必须复用已有 token 级流式输出体验。

要求：

```text
AI 面试官输出实时显示
等待时有思考动画
自动滚动到底部
回答完成后输入框可继续
断流时有兜底提示
```

---

## 43. 数据不足处理

如果用户没有简历、项目或文档，也可以模拟。

但应提示：

```text
如果你上传简历或选择项目，模拟面试会更贴近真实经历。
```

没有材料时可以走：

```text
通用 Java 后端模拟面试
```

---

## 44. 权限和隐私

模拟面试会使用用户私有数据。

必须保证：

```text
只能使用当前用户的简历
只能使用当前用户的项目
只能使用当前用户的文档
不允许前端传 userId
不打印完整简历
不打印完整文档
不打印完整 prompt
不泄露 API Key
```

## 47. 验收标准

完成后应满足：

```text
用户可以创建一场模拟面试
用户可以选择面试类型、难度、轮数
用户可以选择项目和简历
AI 能按阶段进行多轮提问
AI 能结合简历 / 项目 / 文档 / 知识库上下文
面试过程支持 token 级流式输出
面试中不展示完整参考答案
面试结束后生成综合报告
报告能指出阶段表现和高风险回答
低质量回答不会虚高评分
用户可以从报告进入下一步训练
不影响已有项目训练、八股训练、简历训练和复盘 Agent
```

## 48. 简历表达方向

模块完成后可以描述为：

```text
设计并实现真实模拟面试模块，基于用户简历、项目档案、用户文档和知识库 RAG 构建多阶段技术面试流程，支持自我介绍、简历项目追问、技术基础、项目深挖和场景设计等阶段，并通过 token 级流式输出模拟真实面试官追问体验。
```

进一步可以描述为：

```text
模拟面试结束后，系统基于评分 Rubric 生成阶段化综合报告，识别高风险回答、项目表达问题和技术短板，并将结果沉淀到能力画像和复盘 Agent 中，形成从专项训练到综合模拟再到长期复盘的闭环。
```

## 49. 总结

真实模拟面试是 CodeCoach AI 从“专项训练平台”升级为“完整面试训练系统”的关键功能。

它把已有模块串起来：

```text
简历
项目
用户文档
知识库
RAG
token 流式输出
评分体系
能力画像
复盘 Agent
```

最终让用户获得更接近真实技术面的训练体验：

```text
不是刷题
不是聊天
而是一场完整的模拟面试
```
