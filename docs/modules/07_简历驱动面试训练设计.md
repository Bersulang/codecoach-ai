# 简历驱动面试训练设计

## 1. 文档说明

本文档用于设计 CodeCoach AI 的简历驱动面试训练模块。

当前 CodeCoach AI 已经完成：

```text
项目拷打训练
八股问答训练
知识学习模块
能力画像模块
RAG 基础能力
知识文章 RAG
项目档案 RAG
用户文档上传
用户文档 RAG 索引
项目训练接入 USER_UPLOAD RAG
token 级流式训练体验
```

接下来需要在用户文档能力之上，继续构建更有特色、更有面试展示价值、也真正有用户体验价值的模块：

```text
简历驱动面试训练
```

这个模块不是简单的“上传简历后让 AI 总结一下”，也不是普通的“AI 改简历”。

它的核心目标是：

```text
让简历成为 AI 面试训练的起点，让系统像真实面试官一样围绕简历内容、项目经历、技术栈、表述风险点持续追问，并帮助用户验证自己简历上写的内容是否真的讲得清楚、扛得住追问。
```

---

## 2. 模块定位

简历驱动面试训练模块的定位是：

```text
面向 Java 后端求职场景的简历上下文理解、风险识别与面试追问增强模块。
```

它服务于：

```text
简历项目拷打
简历风险点识别
项目经历表达优化
面试前简历复盘
能力画像补充
后续 Agent 面试复盘
```

它不是：

```text
通用简历制作工具
简历模板网站
HR 筛选系统
ATS 打分系统
简历代写工具
求职包装工具
```

CodeCoach AI 的重点不是帮用户“写得更漂亮”，而是帮用户回答：

```text
我简历上写的东西，面试官会怎么追问？
我能不能讲清楚？
我哪里写得太虚？
哪些技术点最容易被深挖？
我的项目经历和真实能力是否匹配？
```

---

## 3. 为什么需要简历驱动训练

当前系统已经有项目档案、用户文档和 RAG 能力，但简历有特殊价值。

真实面试中，面试官通常不是随机提问，而是从简历开始。

面试官会关注：

```text
你写了什么项目
你用了哪些技术
你负责了什么模块
你有没有量化结果
你说的难点是不是真的难
你写的技术栈是否能解释清楚
你描述的项目规模是否可信
你有没有把团队成果包装成个人成果
```

用户最容易出现的问题是：

```text
简历写得很丰富，但讲不清楚
技术栈列得很多，但细节答不上来
项目亮点看起来高级，但缺少真实落地细节
用了“高并发”“分布式”“RAG”“MQ”等词，但没有准备好追问
项目描述和实际代码 / 项目档案不一致
能背八股，但不能结合简历项目表达
```

因此，简历驱动训练的核心价值是：

```text
把简历从静态文档变成动态面试训练入口。
```

---

## 4. 产品核心闭环

简历驱动训练的目标闭环：

```text
用户上传简历
-> 系统解析简历
-> RAG 索引简历内容
-> AI 结构化分析简历
-> 识别项目经历、技术栈、风险点、可追问点
-> 用户选择某个项目或风险点开始训练
-> AI 基于简历 + 项目档案 + 用户文档进行追问
-> token 级流式对话
-> 生成训练报告
-> 沉淀能力画像
-> 推荐下一步学习和训练内容
```

这个闭环使 CodeCoach AI 不只是一个问答练习工具，而是更像一个：

```text
围绕用户真实求职材料构建的 AI 面试训练系统。
```

---

## 5. 模块特色

该模块需要做出特色，而不是普通“AI 简历分析”。

特色一：

```text
简历风险点不是泛泛评价，而是转化为真实面试追问。
```

特色二：

```text
简历项目经历可以和项目档案、README、设计文档进行关联。
```

特色三：

```text
分析结果不是终点，而是训练入口。
```

特色四：

```text
简历中的每个技术关键词都可以变成追问链路。
```

特色五：

```text
训练结果反过来沉淀到能力画像，形成长期成长闭环。
```

## 6. 与已有模块的关系

## 6.1 与用户文档模块的关系

简历本质上是用户上传文档的一种。

用户文档模块负责：

```text
上传
OSS 存储
文件元数据
文本解析
RAG 切片
Embedding
Qdrant 索引
权限控制
```

简历模块负责：

```text
标记某个 user_document 为简历
基于简历文本进行结构化分析
提取项目经历
识别风险点
生成追问建议
关联已有项目档案
发起简历驱动训练
```

也就是说：

```text
用户文档模块是基础设施。
简历模块是业务语义层。
```

---

## 6.2 与项目训练的关系

当前项目训练基于：

```text
project 表中的项目档案
USER_UPLOAD 中关联项目的文档
```

简历模块可以进一步提供：

```text
简历中的项目描述
简历中的技术栈
简历中的职责表述
简历中的项目亮点
简历中的风险点
```

第一版不必新建独立的简历训练 session，可以先复用项目训练。

推荐路线：

```text
简历分析
-> 提取项目经历
-> 用户确认关联到已有项目
-> 从简历项目进入项目拷打训练
```

后续再扩展独立的简历面试 session。

---

## 6.3 与 RAG 的关系

简历进入 RAG 后，可以作为用户私有文档被检索。

检索来源包括：

```text
USER_UPLOAD 简历内容
USER_UPLOAD 项目 README / 设计文档
PROJECT 项目档案
KNOWLEDGE_ARTICLE 系统知识文章
历史项目训练报告，后续
```

简历分析时可以使用：

```text
简历文本
+ 项目档案
+ 项目文档
+ 知识文章
+ 历史训练报告
```

项目训练时可以使用：

```text
当前项目档案
+ 简历中对应项目经历
+ 项目 README
+ 设计文档
+ 用户回答
```

---

## 6.4 与能力画像的关系

简历分析和简历训练结果可以补充能力画像。

例如：

```text
简历写了 Redis 分布式锁
训练时用户解释不清锁续期和误删问题
-> 能力画像中 Redis / 分布式锁维度风险上升
```

未来可以形成：

```text
简历风险点
-> 训练表现
-> 能力画像
-> 学习推荐
-> 再训练
```

## 7. 使用场景

## 7.1 场景一：简历风险点分析

用户上传简历后，点击“分析简历”。

系统输出：

```text
简历整体摘要
技能关键词
项目经历列表
高风险表述
可能被追问的问题
表达优化建议
```

用户可以快速知道：

```text
哪些地方写得太虚
哪些地方容易被追问
哪些技术点需要提前准备
哪些项目经历需要补充细节
```

---

## 7.2 场景二：简历项目拷打

用户在简历分析结果中选择某个项目经历。

系统展示：

```text
该项目的技术栈
该项目的职责描述
该项目的风险点
推荐追问方向
```

用户点击：

```text
开始项目拷打
```

系统进入项目训练，AI 基于简历项目描述进行追问。

---

## 7.3 场景三：简历技术栈深挖

简历中出现：

```text
Spring Boot
Redis
MySQL
MQ
Docker
RAG
Qdrant
```

系统识别这些技术关键词，并生成追问建议：

```text
为什么项目中使用 Redis？
如何保证缓存一致性？
为什么选择 Qdrant？
RAG 检索失败怎么降级？
Embedding 维度为什么选 512？
```

这些问题可以直接进入专项训练。

---

## 7.4 场景四：简历与项目档案一致性检查

用户项目档案中写：

```text
技术栈：Spring Boot、MySQL、Redis
```

简历中写：

```text
使用 Spring Cloud、Kafka、Elasticsearch 构建分布式系统
```

系统可以提示：

```text
简历项目技术栈与项目档案存在不一致，需要确认是否真实使用。
```

这类能力非常有面试价值。

---

## 7.5 场景五：面试前简历复盘

用户面试前打开简历训练页。

系统展示：

```text
当前简历风险点
最近简历项目训练结果
高频追问失败点
建议复习内容
```

用户可以快速进行面试前冲刺。

## 8. 数据模型设计策略

这里需要考虑两种方案。

---

## 9. 方案 A：完全复用 user_document

做法：

```text
简历只是 user_document 中的一种文件
不新增简历表
通过 title / fileType / 用户选择识别简历
分析结果临时生成或存在 user_document 的扩展字段中
```

优点：

```text
开发简单
表少
复用上传和 RAG 能力
```

缺点：

```text
简历语义层不清晰
分析结果不方便复用
项目经历无法结构化保存
风险点无法追踪
后续 Agent 和能力画像接入困难
```

结论：

```text
不推荐作为正式方案。
```

---

## 10. 方案 B：新增简历语义层

新增：

```text
resume_profile
resume_analysis
```

做法：

```text
user_document 负责文件基础能力
resume_profile 负责简历档案
resume_analysis 负责 AI 分析结果
```

优点：

```text
简历语义清晰
分析结果可复用
项目经历可结构化
风险点可长期追踪
适合后续训练和 Agent
```

缺点：

```text
开发稍复杂
需要多维护两张表
```

结论：

```text
推荐采用方案 B。
```

## 11. 第一版设计原则

第一版不追求大而全。

原则：

```text
新增简历语义层
复用用户文档上传和 RAG
复用项目训练
不新建独立简历训练 session
不做 ATS 打分
不做简历代写
不做复杂 Agent
```

第一版核心能力：

```text
从用户文档创建简历档案
AI 分析简历
展示项目经历和风险点
支持从简历项目进入项目拷打
```

## 12. 表一：resume_profile

`resume_profile` 表表示用户的一份简历档案。

建议 SQL：

```sql
CREATE TABLE IF NOT EXISTS resume_profile (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '简历档案ID',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    user_document_id BIGINT NOT NULL COMMENT '关联用户文档ID',
    title VARCHAR(255) NOT NULL COMMENT '简历标题',
    status VARCHAR(32) NOT NULL DEFAULT 'PENDING' COMMENT '状态：PENDING / ANALYZED / FAILED',
    target_role VARCHAR(128) DEFAULT NULL COMMENT '目标岗位',
    summary TEXT DEFAULT NULL COMMENT '简历整体摘要',
    extracted_name VARCHAR(128) DEFAULT NULL COMMENT '解析出的姓名',
    extracted_email VARCHAR(255) DEFAULT NULL COMMENT '解析出的邮箱',
    extracted_phone_masked VARCHAR(64) DEFAULT NULL COMMENT '脱敏手机号',
    error_message VARCHAR(512) DEFAULT NULL COMMENT '错误信息',
    is_deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除：0未删除，1已删除',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted_at DATETIME DEFAULT NULL COMMENT '删除时间',
    KEY idx_user_id (user_id),
    KEY idx_user_document_id (user_document_id),
    KEY idx_user_created (user_id, created_at),
    KEY idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='简历档案表';
```

---

## 13. resume_profile 字段说明

| 字段 | 说明 |
|---|---|
| user_id | 简历所属用户 |
| user_document_id | 关联的用户文档 |
| title | 简历标题 |
| status | 分析状态 |
| target_role | 目标岗位 |
| summary | 简历整体摘要 |
| extracted_name | AI 或规则提取的姓名 |
| extracted_email | AI 或规则提取的邮箱 |
| extracted_phone_masked | 脱敏手机号 |
| error_message | 分析失败原因 |
| is_deleted | 逻辑删除 |

---

## 14. status 设计

`resume_profile.status` 取值：

```text
PENDING
ANALYZED
FAILED
```

含义：

```text
PENDING：已创建，等待分析
ANALYZED：分析完成
FAILED：分析失败
```

---

## 15. 表二：resume_analysis

`resume_analysis` 表保存简历分析结果。

建议 SQL：

```sql
CREATE TABLE IF NOT EXISTS resume_analysis (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '简历分析ID',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    resume_profile_id BIGINT NOT NULL COMMENT '简历档案ID',
    skills JSON DEFAULT NULL COMMENT '技能关键词',
    project_experiences JSON DEFAULT NULL COMMENT '项目经历分析',
    risk_points JSON DEFAULT NULL COMMENT '风险点',
    interview_questions JSON DEFAULT NULL COMMENT '推荐追问题',
    optimization_suggestions JSON DEFAULT NULL COMMENT '表达优化建议',
    overall_comment TEXT DEFAULT NULL COMMENT '整体评价',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    KEY idx_user_id (user_id),
    KEY idx_resume_profile_id (resume_profile_id),
    KEY idx_user_created (user_id, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='简历分析表';
```

## 16. resume_analysis 字段说明

|---|---|

## 22. 简历风险点类型

建议风险点类型包括：

```text
TECH_DEPTH
PROJECT_DEPTH
AUTHENTICITY
VAGUE_EXPRESSION
METRIC_MISSING
OVERCLAIM
INCONSISTENCY
HIGH_FREQUENCY_FOLLOW_UP
```

---

## 23. 技术栈风险

类型：

```text
TECH_DEPTH
```

含义：

```text
简历写了某项技术，但项目描述缺少细节，面试中容易被深挖。
```

示例：

```text
写了 Redis，但没有描述缓存一致性、击穿、穿透、雪崩、分布式锁。
```

---

## 24. 项目深度风险

类型：

```text
PROJECT_DEPTH
```

含义：

```text
项目看起来较大，但个人负责部分不清晰。
```

示例：

```text
写了“负责系统架构设计”，但没有说明具体模块和技术决策。
```

---

## 25. 真实性风险

类型：

```text
AUTHENTICITY
```

含义：

```text
简历描述像团队成果，不像个人实际完成内容。
```

示例：

```text
使用“主导”“负责整体系统”等词，但缺少个人贡献证据。
```

---

## 26. 表达模糊风险

类型：

```text
VAGUE_EXPRESSION
```

含义：

```text
使用大量抽象词，但没有具体实现。
```

示例：

```text
优化系统性能、提升用户体验、提高系统稳定性。
```

---

## 27. 指标缺失风险

类型：

```text
METRIC_MISSING
```

含义：

```text
写了优化结果，但没有指标支撑。
```

示例：

```text
将接口响应速度大幅提升。
```

更好的表达应包含：

```text
接口 P95 响应时间从 800ms 降到 300ms。
```

如果没有真实数据，不能编造，应该改成更稳妥的工程表达。

---

## 28. 过度包装风险

类型：

```text
OVERCLAIM
```

含义：

```text
简历使用了明显超出当前项目规模或个人能力的表述。
```

示例：

```text
“支撑千万级并发”
“设计企业级分布式架构”
“主导核心系统重构”
```

对于学生项目或个人项目，这类表述需要非常谨慎。

---

## 29. 不一致风险

类型：

```text
INCONSISTENCY
```

含义：

```text
简历内容和项目档案、上传文档、训练记录存在冲突。
```

示例：

```text
简历写 Kafka，项目档案写 Redis Stream。
```

---

## 30. 高频追问风险

类型：

```text
HIGH_FREQUENCY_FOLLOW_UP
```

含义：

```text
简历中出现面试官高频深挖词。
```

示例：

```text
高并发
分布式
RAG
MQ
事务
缓存一致性
性能优化
限流
熔断
```

## 32. 简历分析 Prompt 原则

Prompt 要求 AI：

1. 只基于用户提供的简历和上下文；
2. 不编造经历；
3. 不夸大能力；
4. 风险点必须具体；
5. 每个风险点要能转化为面试追问；
6. 建议要面向“如何讲清楚”，而不是“如何包装”；
7. 邮箱、手机号等敏感信息要脱敏；
8. 输出稳定 JSON。

## 48. 隐私与权限

简历属于高度私密信息。

必须保证：

1. 用户只能看到自己的 resume_profile；
2. 用户只能分析自己的简历；
3. 用户只能访问自己的 resume_analysis；
4. user_document 必须属于当前用户；
5. 不允许传 userId；
6. userId 必须从登录态获取；
7. 不打印完整简历内容；
8. 不打印手机号、邮箱；
9. 不允许其他用户检索简历 RAG chunk。

## 50. 错误处理

```text

文档不属于当前用户
文档解析失败

errorMessage 控制 512 字以内
用户可以重新分析
不泄露内部堆栈
```

## 58. 总结

简历驱动面试训练是 CodeCoach AI 的特色模块之一。

它不只是让 AI 看简历，而是把简历变成训练入口。

最终目标是：

```text
让用户知道自己简历上每一句话会被怎么问。
让用户提前暴露项目表达风险。
让用户把简历中的技术亮点讲清楚、讲可信。
让训练结果继续沉淀到能力画像和学习推荐中。
```

第一版应该聚焦：

```text
简历档案
AI 分析
风险点
追问题
项目拷打入口
```

暂时不要做复杂简历编辑、ATS 打分或独立 Agent。

这能让 CodeCoach AI 同时具备：

```text
面试展示价值
工程实现亮点
真实用户体验价值
```

1. EVAL-001：评分 Rubric 与反馈解释体系
2. PLAN-001：AI 面试训练计划 / 7 天冲刺计划
3. AGENT-DOC-001：面试复盘 Agent 设计
4. MOCK-001：真实模拟面试模式
5. OBS-001：AI / RAG 可观测性面板
