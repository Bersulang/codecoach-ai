# TOOL-001 统一 Tool / Action 调用协议验收记录

## 1. 验收结论

TOOL-001 已完成第一阶段 Agent 技术底座升级：

```text
Guide Agent action card
  -> Tool Registry 生成受控 ToolAction
  -> 前端统一调用 /api/agent/tools/execute
  -> 后端校验登录态、Tool 白名单、风险等级、确认状态和对象权限
  -> 执行业务能力或安全跳转
  -> 写入 agent_tool_trace
```

本阶段没有实现 ReAct、MCP、Multi-Agent，也没有开放 HIGH 风险 Tool。


### 3.2 EXECUTE_AFTER_CONFIRM

这些 Tool 会创建资源或触发 AI/业务动作，前端必须弹确认，后端也必须收到 `confirmed=true` 才执行：

```text

START_MOCK_INTERVIEW

```

### 3.3 AUTO_EXECUTE

这些 Tool 只查询摘要或检索数据，不修改业务：


## 4. 前端 Guide 如何调用统一 execute 接口

前端实现位置：

```text
frontend/src/api/agentTool.ts
frontend/src/components/guide/GuideWidget.tsx
frontend/src/types/guide.ts
```

调用流程：

```text
1. Guide 调 /api/guide/chat 获取 answer + actions。
2. action card 内包含 toolName/actionType、toolType、riskLevel、executionMode、requiresConfirmation、targetPath、params。
3. GuideWidget 点击 action 时先用本地 Tool 名白名单判断是否认识该 Tool。
4. 如果 requiresConfirmation=true，使用 Ant Design Modal.confirm 二次确认。
5. 前端调用 POST /api/agent/tools/execute。
6. 请求体只传 toolName、agentType、confirmed、params，不传 userId。
7. 后端返回 ToolExecuteResult。
8. 如果 success=false，前端展示友好错误。
9. 如果 success=true 且有 targetPath，前端使用 React Router navigate 跳转。
10. 如果 displayType=SUMMARY，可把 message 作为 Guide assistant 消息展示。
```

统一执行请求示例：

```json
{
  "toolName": "START_QUESTION_TRAINING",
  "agentType": "GUIDE",
  "confirmed": true,
  "params": {}
}
```

## 5. Trace 记录字段

Trace 表：

```text
agent_tool_trace
```

字段：

| 字段 | 说明 |
|---|---|
| id | 自增主键 |
| trace_id | 单次 Tool 调用 Trace ID |
| user_id | 当前登录用户 ID，来自登录态 |
| agent_type | 调用来源，例如 GUIDE |
| tool_name | Tool 名称 |
| tool_type | NAVIGATION / QUERY / COMMAND |
| input_summary | 脱敏后的输入摘要 |
| output_summary | 脱敏后的输出摘要 |
| success | 是否成功 |
| error_code | 错误码 |
| latency_ms | 执行耗时 |
| created_at | 创建时间 |

脱敏策略：

```text
key/token/content/document/resume/answer 等敏感字段不会完整落库；
输出只记录 success/message/targetPath/errorCode/displayType/dataKeys；
不记录完整简历、完整文档、完整用户回答、API Key。
```

## 6. 未知 Tool / HIGH Tool 防护

### 6.1 未知 Tool

后端 Registry 按 `toolName` 精确匹配：

```text
AgentToolRegistry.requireTool(toolName)
```

如果 Tool 不存在：

```text
返回“未知 Tool，已拒绝执行”
不会进入业务 Service
不会信任前端 targetPath
```

前端也有一层 Tool 名白名单：

```text
GuideWidget.isAllowedAction()
```

前端未知 Tool 不展示、不执行。但真正安全边界仍然后端 Registry。

### 6.2 HIGH Tool

第一版没有注册 HIGH 风险 Tool。

双重防护：

```text
1. AgentToolRegistry 构造时过滤 riskLevel=HIGH 的 Tool；
2. AgentToolExecutor 执行前再次判断 riskLevel=HIGH 并拒绝。
```

因此即使后续误加 HIGH Tool Bean，也不会被 Guide/Agent 可用列表暴露，执行入口也会拒绝。

## 7. 越权访问防护

### 7.1 userId 来源

Tool 执行时：

```text
Long userId = UserContext.getCurrentUserId()
```

前端不能传 userId，后端也不读取请求体 userId。

### 7.2 项目权限

`START_PROJECT_TRAINING`：

```text
如果传 projectId：
  查询 project；
  校验 project 存在、未删除；
  校验 project.userId == 当前 userId；
否则：
  只从当前 userId 的项目中选择最近项目。
```

业务层 `InterviewSessionService.createSession()` 也会再次校验项目归属。

### 7.3 简历权限

`ANALYZE_RESUME` / `START_MOCK_INTERVIEW` 可选 resumeId：

```text
查询 resume；
校验 resume 存在、未删除；
校验 resume.userId == 当前 userId。
```

业务层 `ResumeService.analyze()`、模拟面试 Service 也会再次校验归属。

### 7.4 用户文档权限

`SEARCH_USER_DOCUMENTS`：

```text
Tool 固定 sourceTypes=USER_UPLOAD；
RagRetrievalService 内部从 UserContext 获取当前用户；
检索过滤 userId；
结果转 chunk 时再次检查文档可读性。
```

前端不能通过 params 覆盖 userId。

### 7.5 知识文章权限

`SEARCH_KNOWLEDGE`：

```text
Tool 固定 sourceTypes=KNOWLEDGE_ARTICLE；
只检索系统知识库文章；
不会读取用户私有文档。
```

## 10. 当前限制

1. 第一版 Tool Registry 是 Spring Bean 静态注册，不支持管理端动态启停。
2. Tool 参数校验主要是轻量校验，尚未引入 JSON Schema 或 Bean Validation per Tool。
3. Guide 仍使用 JSON action intent，不是模型原生 function calling / tool_calls。
4. `CREATE_PROJECT_FROM_RESUME` 目前只做 SUGGEST_ONLY 跳转，不自动保存项目档案。
5. Query Tool 目前返回的是摘要数据，不面向复杂可视化展示。
6. Trace 已记录 Tool 调用摘要，但还没有完整 Agent Run / Step / Observation 分层。
7. 没有实现 ReAct、Plan-Execute、MCP、Multi-Agent。
8. HIGH 风险 Tool 第一版完全不开放。

---

## 11. 下一步升级方向

1. 引入 `AgentRun` / `AgentStep` / `ToolObservation`，把单次 Tool Trace 升级为完整 Agent Trace。
2. 给 Tool 参数增加 JSON Schema，用于前后端表单、校验和模型 tool schema 生成。
3. 将 Guide 的 JSON action intent 迁移到模型原生 `tool_calls`。
4. 增加 Tool 权限策略对象，例如 `requiresLogin`、`objectScope`、`allowedRoles`、`riskPolicy`。
5. 为 `CREATE_PROJECT_FROM_RESUME` 增加明确确认、草稿预览、用户确认保存流程。
6. 为 Query Tool 增加更适合前端展示的 `displayType`，例如 `LIST`、`CHART`、`EVIDENCE_CARD`。
7. 增加 HIGH 风险 Tool 的人工确认和不可逆操作保护，但默认不对 Agent 开放。
8. 增加 Tool 调用看板，按 toolName、success、latency、errorCode 观察 Agent 行为质量。
9. 后续在复盘 Agent、模拟面试 Agent 中复用同一 Tool Registry。
