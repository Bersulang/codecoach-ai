# AI 能力模块设计

## 1. 文档说明

本文档用于设计 CodeCoach AI 的 AI 能力模块。

当前项目已经完成 MVP 阶段，能够通过真实大模型 API 完成：

```text
生成第一道项目面试题
生成回答反馈和下一轮追问
生成训练报告
```

但当前 AI 能力仍处于初级阶段，更像是业务代码直接调用大模型接口。

接下来需要将 AI 能力升级为可维护、可扩展、可观测的 AI 工程模块，为后续八股问答、知识学习、能力画像和 RAG 个性化增强提供基础能力。

---

## 2. 模块目标

AI 能力模块的目标是：

```text
为 CodeCoach AI 提供统一、稳定、可配置、可观测的大模型调用能力。
```

具体目标：

1. 将 Prompt 从业务代码中抽离；
2. 建立 Prompt 模板体系；
3. 支持 Prompt 变量注入；
4. 支持结构化 JSON 输出解析；
5. 支持 AI 调用日志记录；
6. 支持多模型配置和切换；
7. 支持不同训练场景复用 AI 能力；
8. 支持异常处理和错误提示；
9. 为后续 RAG 接入预留扩展点。

---

## 3. 当前问题

当前 AI 相关能力存在以下问题：

### 3.1 Prompt 可维护性不足

如果 Prompt 直接写在 Java 代码中，会导致：

1. Prompt 修改需要改代码；
2. Prompt 版本难以管理；
3. Prompt 内容不方便调试；
4. 不同 AI 任务的 Prompt 容易混在一起；
5. 后续八股问答、知识学习、RAG 无法复用。

---

### 3.2 模型输出不稳定

真实大模型可能返回：

1. 纯文本；
2. Markdown；
3. JSON 代码块；
4. JSON 前后带解释；
5. 字段缺失；
6. 格式错误；
7. 空内容；
8. 非预期结构。

如果没有统一解析和校验，容易导致：

1. 后端解析异常；
2. 前端页面崩溃；
3. 报告字段为空；
4. 用户体验不稳定。

---

### 3.3 AI 调用不可观测

如果没有日志记录，就无法回答：

1. 哪次 AI 调用失败了？
2. 调用耗时多久？
3. 使用了哪个模型？
4. 哪类任务最容易失败？
5. 模型返回了什么？
6. 是否存在重复调用？
7. token 成本如何？

---

### 3.4 模型供应商耦合风险

如果业务代码直接依赖某个模型厂商，会导致：

1. 后续切换模型困难；
2. 测试环境不方便；
3. 多模型策略难以实现；
4. 模型异常时影响整个系统。

---

### 3.5 训练策略不够清晰

当前 AI 训练主要关注能否生成问题和反馈，但后续需要支持：

1. 不同难度；
2. 不同岗位；
3. 不同面试风格；
4. 不同知识点；
5. 不同训练任务类型。

因此需要统一的 AI 任务抽象。

---

## 4. 设计原则

AI 能力模块遵循以下原则：

### 4.1 业务层不直接依赖具体模型

业务模块只依赖接口：

```text
AiInterviewService
```

不直接依赖：

```text
MiMo API
OpenAI API
DeepSeek API
```

---

### 4.2 Prompt 模板资源化

Prompt 不直接写死在 Java 业务逻辑中，而是存放在资源文件中。

建议目录：

```text
backend/src/main/resources/prompts
├── interview_first_question.md
├── interview_feedback_next_question.md
└── interview_report.md
```

后续可扩展：

```text
question_practice_first_question.md
question_practice_feedback.md
knowledge_explain.md
rag_interview_context.md
```

---

### 4.3 真实模型失败不 fallback Mock

生产 / 真实模型模式下，如果 AI 调用失败，应返回：

```json
{
  "code": 3003,
  "message": "AI 调用失败，请稍后重试",
  "data": null
}
```

不能 fallback 到 Mock。

原因：

1. Mock 内容会让用户误以为产品质量很差；
2. 真实失败应明确暴露；
3. Mock 仅用于本地开发和测试。

---

### 4.4 结构化输出必须校验

如果要求模型返回 JSON，后端必须：

1. 提取 JSON；
2. 解析 JSON；
3. 校验字段；
4. 处理异常；
5. 记录日志；
6. 返回明确错误。

---

### 4.5 AI 调用必须可观测

所有关键 AI 调用都应记录日志。

至少记录：

1. 请求类型；
2. 模型；
3. 耗时；
4. 是否成功；
5. 错误信息；
6. 用户和会话关联信息。

---

### 4.6 API Key 不进入 Git

API Key 必须来自：

1. 本地忽略配置文件；
2. 环境变量；
3. 服务器 `.env`；
4. 密钥管理服务，后续。

不能提交到 Git。

## 5. AI 服务架构

### 5.1 当前核心接口

当前项目已有或应保留：

```java
public interface AiInterviewService {

    String generateFirstQuestion(Project project, String targetRole, String difficulty);

    FeedbackAndQuestionResult generateFeedbackAndNextQuestion(InterviewContext context);

    ReportGenerateResult generateReport(InterviewContext context);
}
```

---

### 5.2 实现类规划

建议结构：

```text
AiInterviewService
├── OpenAiCompatibleAiInterviewServiceImpl
└── MockAiInterviewServiceImpl
```

说明：

| 实现 | 用途 |
|---|---|
| OpenAiCompatibleAiInterviewServiceImpl | 真实模型调用 |
| MockAiInterviewServiceImpl | 本地测试 / mock profile |

要求：

1. 真实模式不 fallback Mock；
2. Mock 不能在线上启用；
3. 通过配置选择 provider；
4. 后续可扩展更多 provider。

---

### 5.3 配置设计

建议配置：

```yaml
ai:
  provider: openai-compatible
  openai-compatible:
    base-url: ${AI_BASE_URL:}
    api-key: ${AI_API_KEY:}
    model: ${AI_MODEL:}
    timeout-seconds: 120
```

本地开发可创建：

```text
backend/src/main/resources/application-local.yml
```

内容示例：

```yaml
ai:
  provider: openai-compatible
  openai-compatible:
    base-url: https://your-api-base-url
    api-key: your-api-key
    model: your-model-name
    timeout-seconds: 120
```

注意：

```text
application-local.yml 必须被 .gitignore 忽略。
```

### 6.4 第一题 Prompt

文件：

```text
interview_first_question.md
```

目标：

```text
基于项目档案生成第一道面试问题。
```

输出要求：

```text
只输出一道面试问题，不输出 JSON，不输出解释。
```

输入变量：

1. 项目名称；
2. 项目描述；
3. 技术栈；
4. 负责模块；
5. 项目亮点；
6. 项目难点；
7. 目标岗位；
8. 难度。

---

### 6.5 反馈和追问 Prompt

文件：

```text
interview_feedback_next_question.md
```

目标：

```text
基于用户回答生成反馈和下一轮追问。
```

输出格式：

```json
{
  "feedback": "对用户本轮回答的评价",
  "nextQuestion": "下一轮追问"
}
```

要求：

1. 必须输出 JSON；
2. 不输出 Markdown；
3. 不输出解释；
4. feedback 要具体；
5. nextQuestion 要基于用户回答继续追问；
6. 追问应符合当前难度。

---

### 6.6 报告 Prompt

文件：

```text
interview_report.md
```

目标：

```text
基于完整训练记录生成训练报告。
```

输出格式：

```json
{
  "totalScore": 78,
  "summary": "总体评价",
  "strengths": ["优点1", "优点2"],
  "weaknesses": ["薄弱点1", "薄弱点2"],
  "suggestions": ["建议1", "建议2"],
  "qaReview": [
    {
      "question": "问题",
      "answer": "回答",
      "feedback": "反馈"
    }
  ]
}
```

要求：

1. totalScore 范围 0-100；
2. summary 要具体；
3. strengths 不少于 1 条；
4. weaknesses 不少于 1 条；
5. suggestions 不少于 1 条；
6. qaReview 尽量覆盖用户回答过的轮次；
7. 必须输出合法 JSON。

## 7. 结构化输出解析

### 7.1 AiJsonParser

建议新增：

```text
AiJsonParser
```

职责：

1. 清理模型输出；
2. 提取 Markdown JSON 代码块；
3. 提取第一个 JSON 对象；
4. 使用 Jackson 解析；
5. 处理解析异常；
6. 返回目标类型对象。

---

### 7.2 清理规则

模型可能返回：

```text
```json
{...}
```
```

或者：

```text
好的，以下是结果：
{...}
```

解析器需要支持：

1. 去掉 ```json；
2. 去掉 ```；
3. 提取第一个 `{` 到最后一个 `}`；
4. trim 空白；
5. Jackson 解析。

---

### 7.3 结构校验

解析后还要校验字段。

FeedbackAndQuestionResult：

1. feedback 不能为空；
2. nextQuestion 非最后一轮时不能为空。

ReportGenerateResult：

1. totalScore 不为空；
2. totalScore 在 0-100；
3. summary 不为空；
4. strengths 不为空；
5. weaknesses 不为空；
6. suggestions 不为空；
7. qaReview 可以为空，但必须是数组结构。

---

### 7.4 失败处理

解析失败时：

1. 记录原始模型输出；
2. 记录错误原因；
3. 抛出 BusinessException；
4. 返回 code=3003；
5. 不 fallback Mock。

---

## 8. AI 调用日志设计

### 8.1 目标

AI 调用日志用于：

1. 排查模型失败；
2. 观察调用耗时；
3. 统计调用成功率；
4. 分析 token 成本；
5. 支撑后续管理后台；
6. 防止重复调用造成成本浪费。

---

### 8.2 日志表

当前已有：

```text
ai_call_log
```

如字段不足，后续可演进。

建议字段：

| 字段 | 说明 |
|---|---|
| id | 主键 |
| user_id | 用户 ID |
| session_id | 训练会话 ID |
| provider | 模型提供方 |
| model_name | 模型名称 |
| request_type | 调用类型 |
| prompt_tokens | 输入 token |
| completion_tokens | 输出 token |
| total_tokens | 总 token |
| latency_ms | 耗时 |
| success | 是否成功 |
| error_message | 错误信息 |
| raw_response | 原始响应，可选 |
| created_at | 创建时间 |

---

### 8.3 request_type 枚举

项目面试阶段：

```text
INTERVIEW_FIRST_QUESTION
INTERVIEW_FEEDBACK_NEXT_QUESTION
INTERVIEW_REPORT
```

后续扩展：

```text
QUESTION_PRACTICE
KNOWLEDGE_EXPLAIN
RAG_CONTEXT_REWRITE
ABILITY_ANALYSIS
```

---

### 8.4 记录策略

每次真实 AI 调用都要记录：

1. 成功也记录；
2. 失败也记录；
3. 不记录 API Key；
4. rawRequest 谨慎记录，避免敏感信息；
5. rawResponse 可以按需记录，后续可配置开关。

---

## 9. 错误处理策略

### 9.1 通用错误码

AI 相关错误：

| code | message |
|---|---|
| 3003 | AI 调用失败，请稍后重试 |
| 3004 | 当前回答正在处理中，请稍后 |

---

### 9.2 常见异常

| 异常 | 处理 |
|---|---|
| API Key 缺失 | 返回 3003，记录配置错误 |
| baseUrl 缺失 | 返回 3003，记录配置错误 |
| model 缺失 | 返回 3003，记录配置错误 |
| HTTP 超时 | 返回 3003 |
| HTTP 401 / 403 | 返回 3003，日志记录鉴权失败 |
| HTTP 429 | 返回 3003，日志记录限流 |
| 模型返回空内容 | 返回 3003 |
| JSON 解析失败 | 返回 3003 |
| 字段校验失败 | 返回 3003 |

---

### 9.3 前端提示

前端统一展示：

```text
AI 调用失败，请稍后重试。
```

避免展示：

1. HTTP 500；
2. JSON parse error；
3. API key invalid；
4. stack trace；
5. 模型原始错误。

---

## 10. Mock 与真实模型切换策略

### 10.1 Mock 使用场景

Mock 仅用于：

1. 本地无 API Key 开发；
2. 自动化测试；
3. 前端联调；
4. AI 服务不可用时的开发模拟。

---

### 10.2 Mock 禁止场景

Mock 不应用于：

1. 生产环境；
2. 真实用户训练；
3. 真实 AI 调用失败兜底；
4. 上线演示的默认模式。

---

### 10.3 切换方式

建议通过 profile 或 property 控制：

```yaml
ai:
  provider: mock
```

或：

```yaml
ai:
  provider: openai-compatible
```

要求：

1. 同一时间只能有一个 AiInterviewService 生效；
2. provider 配置错误时启动失败或返回明确错误；
3. 避免 Spring 注入多个实现冲突。

## 11. 上下文构造设计

### 11.1 InterviewContext

InterviewContext 应包含：

1. sessionId；
2. userId；
3. project；
4. targetRole；
5. difficulty；
6. currentRound；
7. maxRound；
8. historyMessages；
9. userAnswer。

---

### 11.2 历史消息格式

传给模型的历史消息不应直接使用数据库实体。

建议转换为文本：

```text
第 1 轮
AI 问题：...
用户回答：...
AI 反馈：...
AI 追问：...

第 2 轮
AI 问题：...
用户回答：...
AI 反馈：...
```

这样模型更容易理解上下文。

---

### 11.3 上下文长度控制

后续需要考虑：

1. 历史消息过长截断；
2. 只保留最近 N 轮；
3. 项目档案摘要化；
4. RAG 检索内容长度限制；
5. Prompt token 成本控制。

MVP 阶段可以先不做复杂截断，但需要预留设计。

## 12. 安全注意事项

### 12.1 API Key 安全

要求：

1. 不写死在代码；
2. 不提交 Git；
3. 不打印到日志；
4. 不返回给前端；
5. 部署时通过环境变量注入。

---

### 12.2 用户数据安全

AI Prompt 中可能包含用户项目经历，应注意：

1. 不记录过多敏感原始 Prompt；
2. rawResponse 记录应可配置；
3. 后续支持用户删除数据；
4. 用户数据必须按 userId 隔离。

---

### 12.3 日志脱敏

日志中不能输出：

1. API Key；
2. Authorization Header；
3. 用户密码；
4. 过长 Prompt，除非明确开启 debug。
