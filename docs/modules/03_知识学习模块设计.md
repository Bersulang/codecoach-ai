# 知识学习模块设计

## 1. 文档说明

本文档用于设计 CodeCoach AI 的知识学习模块。

当前 CodeCoach AI 已经完成：

```text
项目拷打训练
八股问答训练
训练报告
训练历史
AI Prompt 模板体系
AI 调用日志
官网与产品化前端体验
```

当前产品已经具备：

```text
训练 -> 报告
```

的核心闭环。

但对于一个真正的 AI 面试训练产品来说，仅仅告诉用户“你哪里薄弱”还不够。用户还需要知道：

```text
我应该去哪里学习？
我应该怎么补？
我学完后如何再次训练？
```

因此，下一阶段需要补充知识学习模块，让产品闭环升级为：

```text
训练 -> 报告 -> 学习 -> 再训练
```

---

## 2. 模块定位

知识学习模块不是普通知识库，也不是 JavaGuide、小林 Coding 等资料站点的复制品。

它的定位是：

```text
面向 Java 后端面试训练场景的面试表达型知识学习模块。
```

普通知识库更关注：

```text
知识解释是否全面
内容体系是否完整
是否适合系统学习
是否覆盖所有细节
```

CodeCoach AI 的知识学习模块更关注：

```text
面试中怎么讲
被追问时怎么答
如何结合项目场景表达
常见错误表达是什么
学完后如何再次训练
```

因此，它不是“大而全教程”，而是服务于训练闭环的学习模块。

---

## 3. 核心目标

知识学习模块的核心目标是：

```text
让用户在训练报告中发现薄弱点后，可以进入对应知识点进行学习，并再次发起专项训练。
```

具体目标：

1. 将八股训练报告中的薄弱点和知识点关联；
2. 为每个高频知识点提供面试表达型学习内容；
3. 帮助用户形成标准回答结构；
4. 提供常见追问和易错点；
5. 提供项目表达模板；
6. 支持从知识文章直接发起专项八股训练；
7. 为后续能力画像、复习计划和 RAG 扩展打基础。

---

## 4. 目标用户

知识学习模块主要面向：

```text
Java 后端实习求职者
Java 后端校招候选人
准备考研复试的学生
初级后端开发者
准备转 Java 后端方向的开发者
```

这些用户通常具备以下特点：

1. 学过相关知识，但表达不稳定；
2. 背过八股，但回答缺少结构；
3. 项目中用过技术，但无法解释选型原因；
4. 被面试官连续追问后容易卡住；
5. 缺少训练后针对薄弱点的学习路径。

## 5. 典型使用场景

### 5.1 八股训练后学习

用户完成一次 Redis 缓存击穿训练。

报告指出：

```text
对互斥锁方案的高并发竞争问题理解不深。
对逻辑过期方案的适用场景表达不完整。
```

用户点击：

```text
学习 Redis 缓存击穿
```

进入知识文章，学习：

1. 缓存击穿的一句话解释；
2. 标准面试回答结构；
3. 互斥锁和逻辑过期方案；
4. 高频追问；
5. 项目表达模板；
6. 再次专项训练入口。

---

### 5.2 项目训练后学习

用户在项目拷打训练中被指出：

```text
Redis 分布式锁的异常场景解释不完整。
```

系统推荐：

```text
Redis 分布式锁
Redisson
幂等性
缓存一致性
```

用户可以进入对应知识点学习，再回到项目训练中优化表达。

---

### 5.3 面试前快速复习

用户面试前想快速复习 MySQL 索引。

进入知识学习页，查看：

```text
MySQL -> 索引
```

重点学习：

1. 一句话解释；
2. B+ 树原理；
3. 回表；
4. 索引失效；
5. 联合索引最左前缀；
6. 常见追问；
7. 面试标准回答。

---

### 5.4 学完后再次训练

用户读完 Redis 缓存击穿文章后，点击：

```text
开始专项训练
```

系统直接基于该 topic 创建八股训练会话。

形成：

```text
学习 -> 训练 -> 报告 -> 再学习
```

的循环。

## 8. 知识内容形态

知识学习模块的内容不应是普通长文，而应该是：

```text
面试表达卡片
```

每篇知识文章建议采用统一结构：

```text
1. 一句话解释
2. 面试标准回答
3. 核心原理
4. 常见追问
5. 易错点
6. 项目表达模板
7. 推荐训练
```

## 9. 知识文章结构规范

每篇 Markdown 知识文章建议遵循以下结构。

```markdown
# 知识点名称

## 1. 一句话解释

用一句话解释该知识点，适合面试开场。

## 2. 面试标准回答

给出结构化回答方式。

推荐结构：

定义
-> 出现原因
-> 核心机制
-> 常见方案
-> 优缺点
-> 项目场景

## 3. 核心原理

解释必须掌握的底层机制。

## 4. 常见追问

列出面试官可能继续追问的问题。

## 5. 易错点

指出候选人常见错误表达。

## 6. 项目表达模板

说明如何结合真实项目表达。

## 7. 推荐训练

提示用户可以基于当前知识点发起专项训练。
```

## 10. 示例：Redis 缓存击穿文章结构

```markdown
# Redis 缓存击穿

## 1. 一句话解释

缓存击穿是指某个热点 key 在过期瞬间，大量并发请求同时打到数据库，导致数据库压力骤增。

## 2. 面试标准回答

可以从以下几个角度回答：

1. 什么是缓存击穿；
2. 它和缓存穿透、缓存雪崩的区别；
3. 为什么热点 key 过期会导致问题；
4. 常见解决方案；
5. 各方案的优缺点；
6. 项目中如何选择方案。

## 3. 核心原理

缓存击穿的核心问题不是缓存不存在，而是热点 key 在某个时间点失效，导致大量并发请求同时穿透缓存访问数据库。

## 4. 常见解决方案

### 4.1 互斥锁

只允许一个线程查询数据库并重建缓存，其他线程等待或重试。

优点：

- 实现简单；
- 能有效保护数据库。

缺点：

- 高并发下可能产生锁竞争；
- 响应时间可能变长；
- 要注意锁超时和死锁问题。

### 4.2 逻辑过期

缓存数据不过物理删除，而是在 value 中存储逻辑过期时间。请求发现数据过期后，异步重建缓存，当前请求仍返回旧数据。

优点：

- 不会让大量请求阻塞；
- 适合热点数据。

缺点：

- 短时间内可能返回旧数据；
- 实现复杂度更高。

## 5. 常见追问

1. 缓存击穿和缓存穿透有什么区别？
2. 为什么互斥锁能解决缓存击穿？
3. 互斥锁方案在高并发下有什么问题？
4. 逻辑过期是否会导致数据不一致？
5. 如果 Redis 宕机怎么办？
6. 热点 key 如何识别？

## 6. 易错点

常见错误包括：

1. 把缓存击穿和缓存穿透混为一谈；
2. 只说加锁，不说明锁超时和锁竞争；
3. 只说逻辑过期，不说明旧数据问题；
4. 不结合业务场景选择方案；
5. 不说明热点 key 如何发现。

## 7. 项目表达模板

在我的项目中，如果某些活动数据或热点商品信息访问量很高，我会优先考虑逻辑过期方案。因为这类数据允许短时间弱一致，并且相比互斥锁，逻辑过期可以避免大量请求阻塞。

如果业务要求强一致，我会考虑互斥锁方案，但需要设置合理的锁超时时间，并结合限流和降级，避免锁竞争导致接口响应变慢。
```

## 11. 内容存储策略

本阶段不建议把完整知识正文直接存入 MySQL。

推荐采用：

```text
MySQL 存文章元数据
Markdown 文件存文章正文
```

原因：

1. Markdown 更适合 Git 管理；
2. 内容修改可以走代码 Review；
3. 版本变化清晰；
4. 适合手动维护和 AI 辅助生成；
5. 后续可以直接用于 RAG 切片；
6. 避免数据库存储大量长文本导致维护困难。

## 13. 数据模型设计

### 13.1 knowledge_article

知识文章表只存元数据，不存正文全文。

建议表结构：

```sql
CREATE TABLE knowledge_article (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '文章ID',
    topic_id BIGINT NOT NULL COMMENT '知识点ID',
    title VARCHAR(128) NOT NULL COMMENT '文章标题',
    summary VARCHAR(512) DEFAULT NULL COMMENT '文章摘要',
    content_path VARCHAR(512) NOT NULL COMMENT 'Markdown正文路径',
    source_type VARCHAR(32) NOT NULL DEFAULT 'ORIGINAL' COMMENT '来源类型：ORIGINAL/AI_ASSISTED/EXTERNAL_REFERENCE',
    source_name VARCHAR(128) DEFAULT NULL COMMENT '来源名称',
    source_url VARCHAR(512) DEFAULT NULL COMMENT '参考链接',
    version VARCHAR(32) NOT NULL DEFAULT 'v1' COMMENT '内容版本',
    status VARCHAR(32) NOT NULL DEFAULT 'PUBLISHED' COMMENT '状态：DRAFT/PUBLISHED/DISABLED',
    sort_order INT NOT NULL DEFAULT 0 COMMENT '排序',
    is_deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted_at DATETIME DEFAULT NULL COMMENT '删除时间',
    KEY idx_topic_id (topic_id),
    KEY idx_status (status),
    KEY idx_sort_order (sort_order),
    KEY idx_topic_status_deleted (topic_id, status, is_deleted)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='知识文章表';
```

---

## 14. 字段说明

| 字段 | 说明 |
|---|---|
| topic_id | 关联 knowledge_topic |
| title | 文章标题 |
| summary | 文章摘要 |
| content_path | Markdown 文件路径 |
| source_type | 内容来源 |
| source_name | 来源名称 |
| source_url | 参考链接 |
| version | 内容版本 |
| status | 发布状态 |
| sort_order | 排序 |
| is_deleted | 逻辑删除 |

---

## 15. source_type 说明

`source_type` 用于标记内容来源。

建议枚举：

```text
ORIGINAL
AI_ASSISTED
EXTERNAL_REFERENCE
```

含义：

```text
ORIGINAL：原创整理内容
AI_ASSISTED：AI 辅助生成后人工整理
EXTERNAL_REFERENCE：基于外部资料参考整理
```

注意：

```text
不直接复制 JavaGuide、小林 Coding 等外部资料全文。
可以参考知识体系，但正文应尽量原创整理。
必要时在 source_url 中保留参考链接。
```

## 17. API 初步规划

### 17.1 查询知识文章列表

```http
GET /api/knowledge-articles
```

查询参数：

```text
topicId，可选
category，可选
keyword，可选
pageNum，可选，默认 1
pageSize，可选，默认 10
```

返回字段：

```text
id
topicId
category
topicName
title
summary
version
status
updatedAt
```

用途：

```text
知识学习首页
文章列表
推荐学习区域
```

---

### 17.2 查询知识文章详情

```http
GET /api/knowledge-articles/{articleId}
```

返回字段：

```text
id
topicId
category
topicName
title
summary
content
version
updatedAt
```

其中：

```text
content 由后端读取 content_path 对应的 Markdown 文件获得。
```

---

### 17.3 根据 topic 查询文章

```http
GET /api/knowledge-topics/{topicId}/article
```

用途：

```text
从八股报告页跳转知识学习
从 topic 卡片进入学习
从训练页推荐学习
```

返回：

```text
如果存在文章，返回文章详情；
如果不存在，返回空或业务错误。
```

## 18. 后端实现规划

推荐新增：

```text
KnowledgeArticle
KnowledgeArticleMapper
KnowledgeArticleService
KnowledgeArticleServiceImpl
KnowledgeArticleController
```

推荐 VO / DTO：

```text
KnowledgeArticlePageRequest
KnowledgeArticleListVO
KnowledgeArticleDetailVO
```

后端核心职责：

1. 查询文章元数据；
2. 关联 knowledge_topic；
3. 读取 Markdown 文件正文；
4. 返回文章详情；
5. 处理文件不存在；
6. 不返回数据库内部字段；
7. 保证未登录不能访问学习内容，或根据产品策略开放访问。

```text
知识学习模块需要登录后访问。
```

原因：

1. 它是训练产品的一部分；
2. 后续要关联学习记录；
3. 便于做个性化推荐。

## 25. 与 RAG 的关系

知识文章使用 Markdown 文件存储，有利于后续 RAG。

后续 RAG 阶段可以：

```text
读取 Markdown
-> 分段
-> 生成 embedding
-> 存入向量库
-> 根据训练问题检索相关片段
-> 增强 Prompt
```

因此当前设计要避免把内容封死在数据库中。

当前阶段只做：

```text
Markdown 文件
+ MySQL 元数据
```

后续再做：

```text
Markdown 切片
+ 向量检索
+ Prompt 增强
```
