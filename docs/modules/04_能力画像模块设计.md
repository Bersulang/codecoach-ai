# 能力画像模块设计

## 1. 文档说明

本文档用于设计 CodeCoach AI 的能力画像与成长洞察模块。

当前 CodeCoach AI 已经完成：

```text
项目拷打训练
八股问答训练
项目训练报告
八股训练报告
训练历史
知识学习模块基础能力
知识文章元数据与 Markdown 内容
官网与前端产品化体验
```

当前产品已经具备较完整的训练闭环：

```text
项目档案 / 知识点
-> AI 训练
-> 训练报告
-> 训练历史
-> 知识学习
-> 再次训练
```

但用户还缺少一个长期视角：

```text
我最近练了什么？
我有没有进步？
我最薄弱的是项目表达，还是八股知识？
我应该优先补 Redis、JVM，还是 MySQL？
我的训练结果是否在持续变好？
```

因此，下一阶段需要设计能力画像模块，将一次次训练报告沉淀成长期成长数据。

---

## 2. 模块定位

能力画像模块不是普通后台数据看板，也不是简单分数统计页。

它的定位是：

```text
面向 Java 后端求职训练场景的成长洞察系统。
```

它的核心作用是：

```text
把单次训练报告转化为长期能力画像，帮助用户理解自己的训练进展和薄弱方向。
```

普通数据看板关注：

```text
访问量
数量统计
表格展示
数据汇总
```

CodeCoach AI 的能力画像关注：

```text
训练表现
知识短板
表达能力
追问应对
成长趋势
下一步训练方向
```

因此，它应该服务于用户的求职训练目标，而不是做成后台 BI。

---

## 3. 核心目标

能力画像模块的核心目标是帮助用户回答以下问题：

1. 我最近完成了多少次训练？
2. 我的平均训练分数是多少？
3. 项目拷打和八股问答哪个更薄弱？
4. 哪些知识分类经常低分？
5. 哪些薄弱点反复出现？
6. 最近几次训练有没有进步？
7. 下一步应该优先练什么？

第一版能力画像不追求复杂智能推荐，而是先完成：

```text
报告数据沉淀
-> 能力快照
-> 聚合统计
-> 成长洞察展示
```

后续再结合 RAG 和知识库检索，实现更智能的学习推荐。

---

## 4. 产品闭环位置

能力画像位于训练闭环之后。

当前闭环：

```text
项目训练 / 八股训练
-> 训练报告
-> 训练历史
```

加入能力画像后：

```text
项目训练 / 八股训练
-> 训练报告
-> 能力快照沉淀
-> 成长洞察
-> 下一步训练建议
```

未来结合知识学习和 RAG 后：

```text
训练
-> 报告
-> 能力画像
-> RAG 检索知识库
-> AI 推荐学习内容
-> 学习
-> 再训练
```

---

## 5. 目标用户

能力画像模块面向：

```text
Java 后端实习求职者
Java 后端校招候选人
准备考研复试的学生
初级后端开发者
准备转 Java 后端方向的开发者
```

这些用户通常关心：

1. 自己是否准备充分；
2. 哪些知识点还不稳；
3. 项目能不能讲清楚；
4. 八股回答是否成体系；
5. 面试前应该优先补什么；
6. 多次训练后是否有提升。

---

## 6. 典型使用场景

### 6.1 查看训练总览

用户进入能力画像页，想快速了解最近训练情况。

页面展示：

```text
总训练次数
项目训练次数
八股训练次数
平均分
最近训练时间
最近分数趋势
```

用户可以快速判断：

```text
最近练得够不够？
整体表现是否稳定？
```

---

### 6.2 查看薄弱知识点

用户连续做了几次 Redis 和 JVM 训练。

能力画像发现：

```text
Redis 缓存一致性多次出现薄弱点
JVM GC 回答得分偏低
JUC 线程池追问表现不稳定
```

系统展示：

```text
高频薄弱点
薄弱知识分类
最近相关报告
```

用户可以知道下一步应该重点复习什么。

---

### 6.3 比较项目训练和八股训练表现

用户项目表达得分较高，但八股问答得分偏低。

能力画像展示：

```text
项目训练平均分：84
八股训练平均分：68
```

用户可以理解：

```text
我项目经历讲得还可以，但基础知识表达需要加强。
```

---

### 6.4 面试前查看风险项

用户面试前进入能力画像页。

系统展示：

```text
最近 7 次训练中，MySQL MVCC、Redis 分布式锁、线程池参数设置多次暴露薄弱点。
```

用户可以优先复习这些内容。

---

### 6.5 后续智能推荐

未来 RAG 完成后，能力画像中的薄弱点可以作为用户侧画像信号。

结合知识库检索后，系统可以推荐：

```text
你最近在 Redis 分布式锁异常场景上表现较弱，建议学习《Redis 分布式锁面试表达指南》，然后进行一次 HARD 难度专项训练。
```

## 8. 能力维度设计

能力维度分为三类：

```text
通用表达能力
项目面试能力
八股知识能力
```

---

## 9. 通用表达能力

通用表达能力适用于项目训练和八股训练。

建议第一版包含：

```text
表达结构化
回答完整性
追问应对能力
技术表达清晰度
```

### 9.1 表达结构化

衡量用户回答是否有清晰结构。

典型表现：

```text
是否能按定义、原因、方案、优缺点、场景表达
是否能避免散乱回答
是否能让面试官快速抓住重点
```

---

### 9.2 回答完整性

衡量用户回答是否覆盖核心要点。

典型表现：

```text
是否遗漏关键机制
是否只回答表面概念
是否能覆盖边界条件
```

---

### 9.3 追问应对能力

衡量用户面对连续追问时是否稳定。

典型表现：

```text
能否在追问中继续展开
是否一被追问就卡住
是否能解释方案背后的取舍
```

---

### 9.4 技术表达清晰度

衡量用户能否用清晰语言表达复杂技术。

典型表现：

```text
是否概念混乱
是否表达过于口语化
是否能准确使用技术术语
```

---

## 10. 项目面试能力

项目训练更关注用户是否能讲清楚自己的项目。

建议第一版能力维度：

```text
项目背景表达
技术选型解释
架构设计理解
异常场景分析
性能与高并发意识
工程权衡能力
```

---

### 10.1 项目背景表达

关注用户是否能讲清楚：

```text
项目解决什么问题
业务场景是什么
自己负责什么模块
项目价值是什么
```

---

### 10.2 技术选型解释

关注用户是否能说明：

```text
为什么使用 Redis
为什么使用 MQ
为什么选择某种数据库设计
为什么这样拆分模块
```

---

### 10.3 架构设计理解

关注用户是否理解：

```text
系统模块边界
调用链路
数据流转
核心业务流程
依赖关系
```

---

### 10.4 异常场景分析

关注用户能否回答：

```text
服务失败怎么办
消息丢失怎么办
缓存不一致怎么办
并发冲突怎么办
数据库压力过大怎么办
```

---

### 10.5 性能与高并发意识

关注用户是否能说明：

```text
瓶颈在哪里
如何优化
如何限流
如何缓存
如何削峰
如何避免热点问题
```

---

### 10.6 工程权衡能力

关注用户是否能说明：

```text
方案优缺点
为什么不用另一种方案
一致性和性能如何取舍
复杂度和收益如何平衡
```

## 13. 第一版实现策略

第一版不做复杂 AI 分析。

推荐采用：

```text
规则聚合
+ 报告字段沉淀
+ 分数趋势统计
```

也就是说：

1. 项目报告生成后沉淀项目能力快照；
2. 八股报告生成后沉淀知识分类能力快照；
3. totalScore 作为基础分数；
4. weaknesses / knowledgeGaps 作为薄弱点文本；
5. difficulty 记录训练强度；
6. createdAt 用于趋势统计；
7. 后续再引入 AI 和 RAG 做智能推荐。

---

## 14. 为什么第一版不直接做 AI 推荐

原因：

1. 当前报告数量可能还不够多；
2. 简单 topicId 推荐不够智能；
3. AI 推荐需要结合知识库检索结果；
4. RAG 完成前推荐内容可能不准确；
5. 过早做智能推荐会增加复杂度。

因此，第一版能力画像只负责：

```text
沉淀数据
展示趋势
暴露薄弱点
提供后续推荐基础
```

真正的智能学习推荐放到 RAG 之后。

## 15. 数据模型设计

建议新增表：

```text
user_ability_snapshot
```

用于存储用户每次报告生成后的能力快照。

---

## 16. user_ability_snapshot 表设计

字段建议：

```sql
CREATE TABLE user_ability_snapshot (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '能力快照ID',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    source_type VARCHAR(32) NOT NULL COMMENT '来源类型：PROJECT_REPORT / QUESTION_REPORT',
    source_id BIGINT NOT NULL COMMENT '来源报告ID',
    dimension_code VARCHAR(64) NOT NULL COMMENT '能力维度编码',
    dimension_name VARCHAR(128) NOT NULL COMMENT '能力维度名称',
    category VARCHAR(64) DEFAULT NULL COMMENT '知识分类或能力分类',
    score INT DEFAULT NULL COMMENT '能力分数',
    difficulty VARCHAR(32) DEFAULT NULL COMMENT '训练难度',
    evidence TEXT DEFAULT NULL COMMENT '证据摘要',
    weakness_tags JSON DEFAULT NULL COMMENT '薄弱点标签',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    KEY idx_user_id (user_id),
    KEY idx_source (source_type, source_id),
    KEY idx_dimension (dimension_code),
    KEY idx_user_dimension_created (user_id, dimension_code, created_at),
    KEY idx_user_created (user_id, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户能力快照表';
```

---

## 17. 字段说明

| 字段 | 说明 |
|---|---|
| user_id | 用户 ID |
| source_type | 数据来源 |
| source_id | 来源报告 ID |
| dimension_code | 能力维度编码 |
| dimension_name | 能力维度名称 |
| category | 知识分类或能力分类 |
| score | 能力分数 |
| difficulty | 训练难度 |
| evidence | 报告中的摘要或证据 |
| weakness_tags | 薄弱点标签 |
| created_at | 快照创建时间 |

---

## 18. source_type 设计

`source_type` 建议包含：

```text
PROJECT_REPORT
QUESTION_REPORT
```

含义：

```text
PROJECT_REPORT：来自项目训练报告
QUESTION_REPORT：来自八股训练报告
```

## 19. 快照沉淀策略

### 19.1 八股报告快照

当生成 `question_training_report` 后，沉淀一条或多条快照。

第一版可以先沉淀一条主快照：

```text
dimension_code = topic.category 映射后的维度
dimension_name = topic.category
category = topic.category
score = report.totalScore
evidence = report.summary
weakness_tags = report.weaknesses + report.knowledgeGaps
```

示例：

```text
topic.category = Redis
dimension_code = REDIS
dimension_name = Redis
score = 76
difficulty = NORMAL
evidence = 对缓存击穿定义掌握较好，但对逻辑过期和高并发锁竞争展开不足。
weakness_tags = ["逻辑过期", "锁竞争", "热点 key"]
```

---

### 19.2 项目报告快照

项目报告不像八股报告有明确 topic。

第一版可以沉淀通用项目能力快照。

建议先沉淀 3 条：

```text
PROJECT_BACKGROUND
TECH_SELECTION
FOLLOW_UP_RESPONSE
```

如果暂时没有细粒度分数，可以都使用 report.totalScore。

evidence 可以使用：

```text
report.summary
```

weakness_tags 可以使用：

```text
report.weaknesses
```

后续再通过 AI 或规则从报告中拆分更细维度。

## 20. 幂等设计

报告生成后可能因为重试或重复调用导致快照重复。

需要保证：

```text
同一个 source_type + source_id + dimension_code 不重复插入
```

建议给 `user_ability_snapshot` 增加唯一索引：

```sql
UNIQUE KEY uk_source_dimension (source_type, source_id, dimension_code)
```

这样可以避免重复沉淀。

如果后续希望支持重新生成报告覆盖快照，可以采用：

```text
先删除旧快照，再插入新快照
```

第一版推荐：

```text
如果已存在，则跳过。
```

---

## 21. 实时聚合 vs 快照沉淀

### 21.1 方案 A：实时聚合

每次查询能力画像时，实时扫描报告表。

优点：

```text
表少
实现简单
没有冗余数据
```

缺点：

```text
查询复杂
趋势统计不方便
后续推荐不方便
扩展性较差
```

---

### 21.2 方案 B：快照沉淀

报告生成后，将关键能力数据写入快照表。

优点：

```text
查询快
适合趋势分析
适合后续推荐
适合 RAG 结合
适合能力维度扩展
```

缺点：

```text
需要维护额外表
需要处理幂等
需要在报告生成流程中插入快照逻辑
```

---

### 21.3 推荐方案

推荐使用：

```text
方案 B：快照沉淀
```

原因：

```text
能力画像是长期成长数据，不应该每次都从原始报告临时解析。
```

## 22. API 初步规划

第一版建议提供以下接口。

---

### 22.1 训练总览

```http
GET /api/insights/overview
```

返回：

```json
{
  "totalTrainingCount": 12,
  "projectTrainingCount": 4,
  "questionTrainingCount": 8,
  "averageScore": 78,
  "recentAverageScore": 82,
  "bestDimension": "Redis",
  "weakestDimension": "JVM",
  "lastTrainingAt": "2026-05-11 16:30:00"
}
```

用途：

```text
/insights 顶部总览
/dashboard 成长洞察卡片
```

---

### 22.2 能力维度

```http
GET /api/insights/ability-dimensions
```

返回：

```json
[
  {
    "dimensionCode": "REDIS",
    "dimensionName": "Redis",
    "category": "八股知识",
    "score": 76,
    "trend": "UP",
    "evidenceCount": 3
  }
]
```

用途：

```text
展示用户各能力维度表现。
```

第一版可以简单计算：

```text
每个维度最近 N 次平均分
```

trend 可以先用：

```text
UP
DOWN
FLAT
UNKNOWN
```

---

### 22.3 高频薄弱点

```http
GET /api/insights/weaknesses
```

返回：

```json
[
  {
    "keyword": "逻辑过期",
    "count": 3,
    "relatedDimension": "Redis",
    "latestEvidence": "对逻辑过期方案的适用场景表达不完整。",
    "latestSourceType": "QUESTION_REPORT",
    "latestSourceId": 8
  }
]
```

用途：

```text
展示反复出现的薄弱点。
```

---

### 22.4 最近训练趋势

```http
GET /api/insights/recent-trend
```

返回：

```json
[
```

## 30. 快照写入失败策略

能力快照属于分析数据，不是主业务数据。

因此推荐：

```text
报告生成成功是主流程
能力快照写入是附加流程
```

如果快照写入失败：

```text
记录 warn 日志
不影响报告返回
不回滚训练报告
```

原因：

1. 用户最关心报告是否生成；
2. 快照可以后续补偿；
3. 避免因为分析数据影响主流程稳定性。

后续可以通过补偿任务重建快照。

## 32. 安全与数据隔离

能力画像是用户私有数据。

所有 `/api/insights/**` 接口必须：

1. 登录后访问；
2. userId 从 UserContext 获取；
3. 只查询当前用户数据；
4. 不允许前端传 userId；
5. 不返回其他用户数据；
6. 不返回内部字段。

---

## 33. 性能考虑

第一版数据量较小，可以直接从 `user_ability_snapshot` 聚合。

需要注意：

1. 按 user_id 建索引；
2. 按 user_id + dimension_code + created_at 建索引；
3. trend 查询限制最近 N 条；
4. weaknesses 聚合限制最近一段时间或最近 N 条；
5. 不一次性返回全部历史快照。

## 37. 总结

能力画像模块是 CodeCoach AI 从“训练工具”升级为“成长系统”的关键一步。

它将原本分散的一次次训练报告沉淀成长期数据，让用户看到：

```text
我练了什么
我哪里薄弱
我有没有进步
我下一步应该关注什么
```

第一版不要追求复杂智能推荐，而应先建立稳定的数据底座：

```text
训练报告
-> 能力快照
-> 聚合洞察
-> 成长展示
```

后续再结合知识学习模块和 RAG，形成更智能的个性化学习路径。
