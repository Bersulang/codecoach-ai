# 八股问答训练模块设计

## 1. 文档说明

本文档用于设计 CodeCoach AI 的八股问答训练模块。

当前 CodeCoach AI 已完成项目拷打训练闭环：

```text
项目档案
-> AI 项目拷打训练
-> 多轮追问
-> 训练报告
-> 训练历史
```

接下来需要新增第二类核心训练场景：

```text
八股问答训练
```

该模块不是普通题库，也不是简单刷题系统，而是面向 Java 后端求职者的 AI 知识点追问训练系统。

八股问答模块的目标是帮助用户围绕 Java 后端高频知识点进行表达训练、追问训练和薄弱点复盘，最终与知识学习模块、能力画像模块形成长期学习闭环。

---

## 2. 模块定位

八股问答模块的产品定位是：

```text
面向 Java 后端求职者的 AI 知识点面试训练系统。
```

它解决的问题不是“用户不知道答案”，而是：

```text
用户知道一些概念，但无法在面试中清晰、结构化、深入地表达。
```

核心价值：

1. 帮助用户围绕高频 Java 后端知识点进行模拟面试；
2. 训练用户的标准表达能力；
3. 训练用户面对连续追问时的应答能力；
4. 通过 AI 反馈发现知识盲区；
5. 通过报告沉淀薄弱点；
6. 后续与知识学习模块联动，实现“练习 -> 反馈 -> 学习 -> 再练习”的闭环。

---

## 3. 目标用户与使用场景

### 3.1 目标用户

核心用户：

```text
准备 Java 后端实习、校招、考研复试或转岗面试的学生和初级开发者。
```

典型特征：

1. 学过 Java 后端基础知识；
2. 知道一些八股概念，但表达不系统；
3. 容易在连续追问中暴露知识断层；
4. 需要针对 JVM、JUC、MySQL、Redis、Spring 等专题进行训练；
5. 希望通过 AI 获取即时反馈和改进建议。

---

### 3.2 使用场景

#### 场景一：专项知识点训练

用户选择：

```text
Redis -> 缓存击穿
```

系统开始围绕缓存击穿提问：

```text
什么是缓存击穿？
如何解决？
互斥锁方案有什么问题？
热点 key 过期如何处理？
如果高并发下锁竞争严重怎么办？
```

---

#### 场景二：面试前快速复习

用户选择：

```text
MySQL -> 索引
```

系统模拟面试官连续追问：

```text
索引为什么能提高查询效率？
B+ 树相比 B 树有什么优势？
什么是回表？
什么情况下索引会失效？
联合索引最左前缀原则是什么？
```

---

#### 场景三：根据薄弱点再次训练

用户在项目训练报告中被指出：

```text
对 Redis 缓存一致性回答不够深入。
```

系统推荐用户进入：

```text
Redis -> 缓存一致性
```

进行专项八股训练。

## 4. 与项目拷打训练模块的区别

| 维度 | 项目拷打训练 | 八股问答训练 |
|---|---|---|
| 训练对象 | 用户自己的项目经历 | Java 后端知识点 |
| 输入来源 | 项目档案 | 知识点目录 |
| 训练重点 | 项目表达、技术选型、场景分析 | 概念理解、标准表达、连续追问 |
| 问题类型 | 围绕项目业务和技术实现 | 围绕知识点原理和面试题 |
| 报告重点 | 项目表达能力、技术深度、场景意识 | 知识掌握程度、表达结构、薄弱概念 |
| 后续联动 | 项目优化、项目表达模板 | 知识学习、错题复习、专项训练 |

两者都属于“训练会话”，但领域不同，不建议强行共用同一套业务表。

## 5. 核心业务流程

八股问答训练第一版流程：

```text
选择知识分类
-> 选择知识主题
-> 选择训练难度
-> 创建训练会话
-> AI 生成第一题
-> 用户回答
-> AI 反馈、评分、参考答案、追问
-> 用户继续回答
-> 达到最大轮次或主动结束
-> 生成训练报告
-> 记录训练历史
```

详细流程：

```text
1. 用户进入八股训练页面；
2. 用户选择知识分类，例如 Redis；
3. 用户选择知识主题，例如 缓存击穿；
4. 用户选择难度 EASY / NORMAL / HARD；
5. 用户点击开始训练；
6. 后端创建 question_training_session；
7. AI 根据 topic 生成第一道问题；
8. 系统保存 AI 问题消息；
9. 前端进入八股训练页；
10. 用户提交回答；
11. 后端保存用户回答；
12. AI 生成本轮反馈、参考答案和下一轮追问；
13. 系统保存 AI 反馈和追问；
14. 更新当前轮次；
15. 最后一轮自动生成报告；
16. 用户进入报告页查看复盘。
```

## 7. 训练难度设计

八股问答模块复用现有难度体系：

```text
EASY
NORMAL
HARD
```

### 7.1 EASY

适合刚学完知识点的用户。

训练特点：

1. 问题更基础；
2. 重点确认概念是否理解；
3. AI 反馈更偏引导；
4. 参考答案更完整；
5. 追问不宜过深。

示例：

```text
什么是缓存击穿？它和缓存穿透有什么区别？
```

---

### 7.2 NORMAL

适合常规实习 / 校招面试。

训练特点：

1. 模拟常规技术面；
2. 关注概念、原理、使用场景；
3. 会追问常见解决方案；
4. 要求用户表达清晰；
5. 反馈指出优缺点。

示例：

```text
缓存击穿可以通过互斥锁解决，那么这种方案在高并发场景下有什么问题？
```

---

### 7.3 HARD

适合深度复习和大厂面试准备。

训练特点：

1. 深挖底层原理；
2. 追问边界条件；
3. 关注性能、并发、故障恢复；
4. 反馈更严格；
5. 评分标准更高。

示例：

```text
如果热点 key 在高并发下频繁过期，你如何设计一个既能避免缓存击穿，又不会造成大量线程阻塞的方案？
```

## 8. 数据模型初步规划

八股问答模块建议独立建表，不与项目训练表强行混用。

原因：

1. 项目训练围绕 project；
2. 八股训练围绕 knowledge_topic；
3. 报告维度不同；
4. 后续知识学习和能力画像需要 topic 关联；
5. 独立建表更清晰。

## 11. AI Prompt 设计方向

八股问答 Prompt 不应复用项目拷打 Prompt，但可以复用 AI 能力底座。

建议新增 Prompt 模板：

```text
backend/src/main/resources/prompts
├── question_first_question.md
├── question_feedback_next_question.md
└── question_report.md
```

---

### 11.1 第一题 Prompt

输入：

1. category；
2. topicName；
3. topicDescription；
4. interviewFocus；
5. targetRole；
6. difficulty。

输出：

```text
一道中文八股面试题。
```

要求：

1. 问题要围绕当前 topic；
2. 不要问项目经历；
3. 不要输出 JSON；
4. 适合当前难度。

---

### 11.2 回答反馈 Prompt

输入：

1. topic；
2. difficulty；
3. 当前问题；
4. 用户回答；
5. 历史问答；
6. 当前轮次。

输出 JSON：

```json
{
  "feedback": "本轮回答反馈",
  "referenceAnswer": "参考答案",
  "nextQuestion": "下一轮追问",
  "score": 78
}
```

要求：

1. feedback 指出优点和不足；
2. referenceAnswer 给出结构化参考答案；
3. nextQuestion 基于当前回答继续追问；
4. score 范围 0-100；
5. 只输出 JSON。

---

### 11.3 报告 Prompt

输出 JSON：

```json
{
  "totalScore": 82,
  "summary": "总体评价",
  "strengths": ["..."],
  "weaknesses": ["..."],
  "suggestions": ["..."],
  "knowledgeGaps": ["..."],
  "qaReview": [
    {
      "question": "问题",
      "answer": "用户回答",
      "referenceAnswer": "参考答案",
      "feedback": "反馈"
    }
  ]
}
```

## 14. 与知识学习模块的关系

八股训练报告中的薄弱点应能跳转到知识学习模块。

示例：

用户在 Redis 缓存击穿训练中报告显示：

```text
薄弱点：互斥锁方案在高并发下的锁竞争问题理解不深。
```

后续可以推荐：

```text
学习：Redis 缓存击穿解决方案
学习：Redisson 分布式锁
训练：Redis 分布式锁专项追问
```

这形成：

```text
训练 -> 报告 -> 学习 -> 再训练
```

的闭环。

## 19. 阶段完成标准
