# 语音输入与简历生成项目档案设计

## 1. 文档说明

本文档用于设计 CodeCoach AI 的两个体验增强能力：

```text
训练语音输入
简历一键生成项目档案
```

当前 CodeCoach AI 已经完成：

```text
项目拷打训练
八股问答训练
真实模拟面试
简历驱动训练
用户文档 RAG
项目档案 RAG
知识文章 RAG
token 级流式输出
产品内 AI 训练向导
面试复盘 Agent
能力画像
```

随着产品能力增强，当前还有两个明显体验阻力：

```text
用户训练时需要手动输入回答
用户创建项目档案需要手动填写大量字段
```

因此需要新增：

```text
语音输入：让回答更接近真实面试
简历生成项目档案：降低用户冷启动成本
```

---

## 2. 功能一：训练语音输入

## 2.1 功能定位

训练语音输入的定位是：

```text
让用户可以像真实面试一样用口头表达回答问题。
```

它不是完整语音通话系统，也不是实时会议系统。

第一版目标是：

```text
用户可以点击麦克风录音，将语音转为文字，再作为训练回答提交。
```

---

## 2.2 为什么需要语音输入

真实面试中，用户是说话回答，而不是打字回答。

当前手动输入存在问题：

```text
长回答输入成本高
用户表达节奏不真实
模拟面试沉浸感弱
用户不愿意写长答案
项目拷打和真实面试体验有差距
```

语音输入可以提升：

```text
训练真实感
回答效率
模拟面试沉浸感
口头表达训练价值
```

---

## 2.3 适用页面

第一版建议接入：

```text
项目拷打训练页
八股问答训练页
真实模拟面试页
```

后续可接入：

```text
产品内 AI 训练向导
复盘 Agent
简历训练问答
```

---

## 2.4 第一版交互流程

推荐第一版流程：

```text
用户点击输入框旁的麦克风按钮
-> 开始录音
-> 用户口头回答
-> 点击停止
-> 系统转写为文本
-> 文本填入输入框
-> 用户可修改
-> 用户点击发送
```

不要第一版就自动提交。

原因：

```text
语音识别可能有误
用户需要确认内容
避免误提交
```

---

## 2.5 语音输入状态

前端需要展示状态：

```text
未录音
正在录音
正在识别
识别成功
识别失败
浏览器不支持
权限被拒绝
```

---

## 2.6 交互细节

录音按钮状态：

```text
默认：麦克风图标
录音中：红点 / 波纹 / 计时
识别中：loading
失败：提示重试
```

识别成功后：

```text
转写文本填入输入框
用户可以继续编辑
```

---

## 2.7 技术路线选择

### 方案 A：浏览器 Web Speech API

优点：

```text
实现快
无需后端 ASR 服务
适合快速验证
```

缺点：

```text
兼容性不稳定
不同浏览器支持不同
识别效果不可控
不适合长期可靠产品
```

### 方案 B：后端 ASR 服务

流程：

```text
前端录音
-> 上传音频
-> 后端调用 ASR
-> 返回文本
```

优点：

```text
更稳定
可控性更强
适合正式产品
可做日志和错误处理
```

缺点：

```text
需要接入语音识别服务
可能产生费用
实现复杂度更高
```

---

## 2.8 推荐实现策略

第一版推荐：

```text
先实现前端 Web Speech API 版本
如果浏览器不支持，则隐藏语音按钮或提示手动输入
```

后续升级：

```text
接入后端 ASR 服务
支持音频上传识别
支持更稳定中文识别
支持完整语音模拟面试
```

如果项目时间允许，也可以直接采用后端 ASR 方案。

---

## 2.9 语音输入边界

第一版不做：

```text
AI 语音播报
实时语音通话
自动检测停顿提交
多人语音面试
语音情绪识别
发音评分
```

---

## 3. 功能二：简历一键生成项目档案

## 3.1 功能定位

简历生成项目档案的定位是：

```text
从用户简历分析结果中提取项目经历，并一键生成结构化项目档案。
```

它不是编造项目，也不是自动包装经历。

它的目标是：

```text
降低用户创建项目档案的成本，让用户能快速进入项目拷打和模拟面试。
```

---

## 3.2 为什么需要这个功能

当前项目训练依赖项目档案。

但用户填写项目档案需要输入：

```text
项目名称
项目描述
技术栈
负责模块
项目亮点
项目难点
项目总结
```

这对新用户来说有成本。

而简历中通常已经包含项目经历。

所以可以把流程变成：

```text
上传简历
-> 分析简历
-> 提取项目经历
-> 一键生成项目档案
-> 用户确认编辑
-> 保存项目
-> 开始项目拷打
```

这样体验更顺。

---

## 3.3 产品价值

该功能可以形成冷启动闭环：

```text
用户上传简历
-> 生成项目档案
-> 进入项目训练
-> 生成报告
-> 沉淀能力画像
```

这能避免用户刚注册后不知道如何开始。

---

## 3.4 用户流程

用户路径一：

```text
进入简历训练页
-> 查看分析结果
-> 找到某个项目经历
-> 点击“一键生成项目档案”
-> 系统生成项目草稿
-> 用户确认和编辑
-> 保存项目档案
-> 开始项目拷打
```

用户路径二：

```text
进入项目页面
-> 点击“从简历导入项目”
-> 选择简历分析结果中的项目
-> 生成项目草稿
-> 保存
```

---

## 3.5 生成内容

从简历项目经历生成项目档案时，应尽量填充：

```text
项目名称
项目简介
技术栈
个人职责
核心模块
项目亮点
项目难点
可追问风险点
```

如果当前 project 表字段有限，则按现有字段映射。

不要强行新增过多字段。

---

## 3.6 生成原则

必须遵守：

```text
只基于简历已有内容和分析结果生成
不能编造用户没有写过的经历
不能虚构指标
不能夸大项目规模
不确定的信息要标注为待补充
```

例如：

```text
简历写了“优化接口性能”
```

不能自动改成：

```text
接口性能提升 80%
```

除非简历中明确写了。

---

## 3.7 项目草稿

建议生成后先进入“项目草稿确认页”或弹窗，而不是直接保存。

原因：

```text
AI 可能理解有误
用户需要确认真实性
用户可以补充细节
```

草稿页展示：

```text
项目名称
项目描述
技术栈
个人职责
亮点
难点
待补充项
```

用户可以：

```text
编辑
保存为项目档案
取消
保存后开始训练
```

---

## 3.8 与简历分析的关系

该功能依赖：

```text
resume_analysis.projectExperiences
```

如果简历还没有分析，应提示：

```text
请先分析简历，再从项目经历生成项目档案。
```

---

## 3.9 与项目训练的关系

生成项目档案后，可以直接提供：

```text
开始项目拷打
开始模拟面试
```

这样用户从简历到训练的路径更短。

---

## 3.10 与 RAG 的关系

生成项目档案后：

```text
项目档案本身可以进入 PROJECT RAG
简历原文仍然作为 USER_UPLOAD RAG
项目训练时可以同时检索 PROJECT + USER_UPLOAD
```

这样 AI 能结合：

```text
结构化项目档案
简历中的项目表述
用户上传的项目文档
```

---

## 3.11 重复项目处理

如果用户已经有相似项目，系统应提示：

```text
可能已存在相似项目，是否仍然创建？
```

匹配依据：

```text
项目名称相似
技术栈重叠
简历分析中的 matchedProjectId
```

第一版可以简单做：

```text
如果项目名完全相同，提示用户确认。
```

---

## 4. 页面规划

## 4.1 简历训练页

在每个项目经历卡片中增加按钮：

```text
生成项目档案
```

按钮状态：

```text
未生成：生成项目档案
已关联项目：查看项目 / 开始训练
生成中：正在生成
```

---

## 4.2 项目页面

在项目列表页增加入口：

```text
从简历导入项目
```

用户点击后：

```text
选择简历
-> 选择项目经历
-> 生成项目草稿
-> 保存
```

第一版如果成本较高，可以只在简历训练页做入口。

---

## 4.3 项目草稿确认

可以使用：

```text
弹窗
抽屉
独立页面
```

推荐：

```text
右侧抽屉或大弹窗
```

展示：

```text
AI 从简历中提取的项目档案草稿
```

提示：

```text
请确认这些内容真实准确，再保存为项目档案。
```

---

## 5. 语音输入与模拟面试的关系

语音输入会显著增强真实模拟面试。

模拟面试页中，用户可以：

```text
点击麦克风
口头回答
转写后确认
发送
```

后续可升级为：

```text
AI 读题
用户语音回答
自动转写
自动提交
生成语音级表达反馈
```

但第一版只做语音转文本。

---

## 6. 安全与隐私

## 6.1 语音输入

如果使用浏览器 Web Speech API：

```text
后端不保存音频
前端只拿到转写文本
```

如果后续使用后端 ASR：

```text
音频上传需要限制大小
不长期保存音频
不打印音频内容
不泄露用户回答
```

---

## 6.2 简历生成项目档案

必须保证：

```text
只能使用当前用户简历
只能生成当前用户项目
不能访问他人 resume_analysis
不能访问他人 project
不能编造简历外经历
```

---

## 7. MVP 范围

第一版做：

```text
训练页语音输入转文本
模拟面试页语音输入转文本
简历项目经历生成项目档案草稿
用户确认后保存项目
保存后可进入项目拷打
```

第一版不做：

```text
AI 语音播报
自动语音提交
后端音频长期保存
语音评分
发音分析
自动创建未确认项目
从简历编造项目经历
```

---

## 8. 验收标准

语音输入：

```text
训练输入框有麦克风按钮
用户可以开始 / 停止语音识别
识别结果填入输入框
用户可以编辑识别结果
用户可以发送
不支持语音时有友好提示
不影响原文本输入
```

简历生成项目档案：

```text
简历项目经历卡片有生成项目档案入口
可以生成项目草稿
草稿内容来自简历分析
用户可以编辑草稿
用户确认后保存为项目
保存后可以开始项目拷打
不会自动编造不存在信息
不会越权访问他人简历
```

---

## 9. 简历表达方向

该能力完成后可以描述为：

```text
为提升 AI 面试训练真实感，项目支持训练页语音输入，用户可通过语音回答问题并自动转写为文本，降低长回答输入成本，使项目拷打和模拟面试更接近真实面试场景。
```

也可以描述为：

```text
设计并实现从简历分析结果一键生成项目档案的能力，将简历中的项目经历转化为结构化项目数据，用户确认后即可进入项目拷打训练，形成从简历上传、项目档案生成到 AI 面试训练的冷启动闭环。
```

---

## 10. 总结

语音输入和简历生成项目档案分别解决两个关键问题：

```text
语音输入解决训练表达方式不真实的问题。
简历生成项目档案解决用户冷启动和项目档案填写成本问题。
```

这两个功能都能明显提升真实体验价值。

它们让 CodeCoach AI 更像一个真正的 AI 面试训练产品，而不是一个需要用户大量手动输入的练习系统。
