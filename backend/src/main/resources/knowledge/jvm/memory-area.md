# JVM 内存区域

## 1. 一句话解释

JVM 内存区域是 Java 程序运行时对内存的职责划分，包括线程私有区域和线程共享区域。

## 2. 面试标准回答

面试时可以先按线程私有和线程共享来回答。线程私有区域包括程序计数器、虚拟机栈和本地方法栈。程序计数器记录当前线程执行到哪条字节码；虚拟机栈保存方法调用栈帧，包括局部变量表、操作数栈、返回地址等；本地方法栈服务于 native 方法。

线程共享区域主要包括堆和方法区。堆是对象实例的主要存放位置，也是垃圾回收最关注的区域。方法区在 HotSpot 中由元空间实现，主要存类元信息、常量、静态变量等。运行时常量池属于方法区的一部分。直接内存不属于 JVM 运行时数据区，但 NIO 会使用，过大也可能导致内存问题。

最后结合异常：堆空间不足可能出现 `OutOfMemoryError: Java heap space`，栈深度过深可能出现 `StackOverflowError`，元空间不足可能出现 `Metaspace` OOM。

## 3. 核心原理

线程私有区域随着线程创建和销毁，不需要垃圾回收器重点管理。每个方法调用都会创建栈帧，方法返回后栈帧出栈。递归过深或栈帧过大会导致栈溢出。

堆被所有线程共享，绝大多数对象都在堆上分配。垃圾回收主要围绕堆进行。方法区存放类级别信息，Java 8 之后元空间使用本地内存，默认不再固定在 JVM 堆内。

直接内存常见于 NIO、Netty 等场景，它减少了堆内复制，但也需要控制容量，否则可能出现堆不高但进程内存很高的问题。

## 4. 常见追问

1. 哪些区域是线程私有的？
2. 堆和栈分别存什么？
3. 方法区和元空间是什么关系？
4. 运行时常量池在哪里？
5. `StackOverflowError` 和 OOM 有什么区别？
6. 为什么直接内存也可能导致 OOM？
7. 对象一定分配在堆上吗？
8. 如何排查线上 OOM？

## 5. 易错点

不要只背区域名称，要说清每个区域存什么、生命周期如何、可能出现什么异常。不要把方法区简单等同于永久代，Java 8 后 HotSpot 使用元空间实现方法区。

回答 OOM 排查时，不要只说“加内存”。更好的表达是先看错误类型，再结合堆 dump、GC 日志、对象占用、线程数和直接内存使用情况定位。

## 6. 项目表达模板

在项目排查内存问题时，我会先根据报错判断区域。如果是堆 OOM，会抓取 heap dump 分析大对象和引用链；如果是栈溢出，会检查递归调用或线程栈设置；如果是元空间问题，会关注动态代理、类加载数量；如果是 Netty 或 NIO 场景，还会检查直接内存上限和 ByteBuffer 释放情况。

## 7. 推荐训练

建议继续训练垃圾回收、类加载机制、G1 垃圾回收器和线上 OOM 排查。JVM 内存区域题要练习“区域划分 -> 存储内容 -> 生命周期 -> 异常排查”。
