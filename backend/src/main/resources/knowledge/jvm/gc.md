# JVM 垃圾回收

## 1. 一句话解释

JVM 垃圾回收是自动识别不再被引用的对象并释放内存的机制，核心目标是在吞吐量和停顿时间之间做平衡。

## 2. 面试标准回答

面试时可以先说对象存活判断。主流 JVM 使用可达性分析，从 GC Roots 出发，能被引用链到达的对象认为存活，不能到达的对象可以回收。GC Roots 常见包括虚拟机栈引用的对象、静态变量引用的对象、常量引用的对象和 native 方法引用的对象。

然后说回收算法。标记清除先标记存活对象再清理垃圾，缺点是内存碎片；复制算法把存活对象复制到另一块空间，适合新生代；标记整理把存活对象向一端移动，适合老年代但成本较高。

分代收集基于大多数对象朝生夕死的假设，把堆分为新生代和老年代。新对象先进入新生代，经历多次 GC 仍存活或对象较大时进入老年代。G1 则按 Region 管理堆，优先回收收益高的区域，并尝试控制停顿时间。

## 3. 核心原理

可达性分析解决“哪些对象可以回收”。垃圾回收算法解决“如何回收”。垃圾收集器则是具体实现，负责在不同业务目标下选择合适策略。

Minor GC 通常发生在新生代空间不足时；Full GC 或 Mixed GC 可能涉及老年代，停顿更明显。线上关注的不只是是否发生 GC，还要看频率、停顿时间、回收后内存是否下降。

G1 的核心思路是把堆拆成多个 Region，维护每个 Region 的回收价值，优先回收垃圾比例高的区域。它适合大堆和低停顿目标，但也需要合理参数和监控。

## 5. 易错点

不要说 JVM 用引用计数判断对象存活，主流 HotSpot 使用可达性分析。不要只背算法名称，要讲清每个算法解决什么问题、带来什么代价。

回答 Full GC 排查时，不要直接说“调大堆”。应该先看对象增长、老年代占用、晋升失败、大对象、内存泄漏和 GC 日志，再决定是否调参。

## 6. 项目表达模板

如果线上出现接口抖动或响应时间尖刺，我会先看监控中的 GC 次数和停顿时间。如果 Full GC 频繁，会结合 GC 日志、堆使用曲线和 heap dump 判断是内存泄漏、对象创建过快还是参数不合理。比如大对象频繁进入老年代，就要优化对象生命周期或分页批量大小，而不是简单加机器。
