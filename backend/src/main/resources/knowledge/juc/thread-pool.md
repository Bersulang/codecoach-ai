# JUC 线程池

## 1. 一句话解释

线程池通过复用线程、控制并发数量和统一管理任务执行，避免频繁创建线程带来的资源开销和系统风险。

## 2. 面试标准回答

面试时可以先说为什么需要线程池：直接创建线程成本高，线程数不可控会导致上下文切换增加、内存占用升高，甚至拖垮服务。线程池把线程复用起来，并通过队列和拒绝策略控制任务流量。

核心参数包括核心线程数、最大线程数、空闲线程存活时间、阻塞队列、线程工厂和拒绝策略。任务提交流程是：如果运行线程数小于核心线程数，创建核心线程；否则进入队列；队列满了且线程数小于最大线程数，创建非核心线程；再满就触发拒绝策略。

队列选择也很重要。无界队列可能导致任务堆积和 OOM；有界队列能控制内存，但需要配合拒绝策略。拒绝策略可以抛异常、调用者执行、丢弃任务或丢弃最旧任务，项目中通常会自定义记录日志和告警。

## 3. 核心原理

线程池本质上是在“执行能力”和“任务堆积”之间做平衡。核心线程数决定稳定处理能力，最大线程数决定突发处理上限，队列决定缓冲能力。参数设置不能只看 CPU 核数，还要看任务类型。

CPU 密集型任务线程数通常接近 CPU 核数；IO 密集型任务因为存在等待，可以设置更高线程数。真实项目还要考虑下游数据库、Redis、第三方接口的承载能力，不能让线程池把压力无限放大。

## 5. 易错点

不要只背参数，要讲任务提交流程。不要说线程池越大越好，线程过多会增加上下文切换，还可能压垮下游。不要使用 `Executors.newFixedThreadPool` 等默认工厂作为生产答案，因为它可能隐藏无界队列或线程数风险。

另一个易错点是多个业务混用一个线程池，导致低优先级任务把高优先级任务阻塞。面试时可以补充线程池隔离。

## 6. 项目表达模板

在项目中，我会根据业务类型拆分线程池。比如发送通知、异步日志、导出任务不会共用同一个线程池。线程池使用有界队列和自定义线程名，便于监控和排查。拒绝策略不会静默丢弃，而是记录日志、打点告警，必要时让调用方降级。参数设置会结合任务耗时、QPS、下游承载能力和压测结果调整。

## 7. 推荐训练

建议继续训练 synchronized、volatile、CAS、AQS 和异步任务设计。线程池题要练习从“参数”讲到“线上治理”，这是区分背诵和项目经验的关键。
