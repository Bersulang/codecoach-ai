# MySQL MVCC

## 1. 一句话解释

MVCC 是 MySQL InnoDB 通过多版本数据和 ReadView 实现并发读写的一种机制，用来减少读写冲突。

## 2. 面试标准回答

面试时可以先说背景：如果所有读写都靠锁，读写互相阻塞，数据库并发能力会很差。MVCC 的作用是让普通查询在很多情况下不加锁也能读到一个一致的历史版本。

核心组成包括隐藏字段、undo log 和 ReadView。InnoDB 每行记录有事务 id 和回滚指针，更新数据时会生成 undo log，形成版本链。快照读执行时会生成 ReadView，用来判断版本链中哪个版本对当前事务可见。

在可重复读隔离级别下，同一个事务中多次快照读通常使用同一个 ReadView，所以能读到一致结果。在读已提交隔离级别下，每次查询都会生成新的 ReadView，所以可能读到其他事务已提交的新数据。

还要区分快照读和当前读。普通 `SELECT` 通常是快照读；`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE` 属于当前读，会读取最新版本并加锁。

## 3. 核心原理

MVCC 的关键是可见性判断。ReadView 里会记录当前活跃事务集合、最小活跃事务 id、下一个事务 id 等信息。读取某行时，InnoDB 沿着版本链判断每个版本是否对当前事务可见，不可见就继续通过 undo log 找更早版本。

MVCC 解决的是快照读的一致性问题，不等于完全不需要锁。当前读仍然需要加锁，范围更新也可能使用临键锁防止幻读。可重复读下，InnoDB 通过 MVCC 加锁机制共同处理幻读问题。

## 4. 常见追问

1. MVCC 解决了什么问题？
2. undo log 在 MVCC 中有什么作用？
3. ReadView 里包含哪些关键信息？
4. 读已提交和可重复读下 ReadView 有什么区别？
5. 快照读和当前读有什么区别？
6. MVCC 能不能解决所有幻读？
7. MVCC 和锁机制是什么关系？
8. 长事务会对 MVCC 造成什么影响？

## 5. 易错点

不要把 MVCC 说成“完全不用锁”。它主要服务于快照读，当前读仍然涉及锁。不要只背版本链和 ReadView 名词，还要说明它们如何配合判断可见性。

另一个常见错误是把可重复读理解成“永远读不到新数据”。当前读读取的是最新版本，并不是快照读的历史版本，这一点经常被追问。

## 6. 项目表达模板

在项目中，如果涉及订单状态、库存记录等并发读写，我会区分快照读和当前读。普通展示类查询依赖 InnoDB 的 MVCC 保证一致性，减少锁冲突；涉及状态变更时，会使用当前读或条件更新，并结合事务边界控制并发。排查慢查询或 undo log 膨胀时，也会关注是否存在长事务导致历史版本无法及时清理。

## 7. 推荐训练

建议继续训练事务隔离级别、锁机制、undo log 和幻读。回答 MVCC 时重点练习“为什么需要、由什么组成、如何判断可见、和锁有什么关系”。
