# MySQL 索引

## 1. 一句话解释

MySQL 索引是帮助数据库快速定位数据的数据结构，InnoDB 中最核心的是基于 B+ 树的索引。

## 2. 面试标准回答

面试时可以从作用讲起：索引的目标是减少扫描行数，提升查询效率。InnoDB 使用 B+ 树组织索引，因为 B+ 树高度低、适合磁盘页读取，并且叶子节点有序链表适合范围查询。

然后说明分类：主键索引的叶子节点存整行数据，二级索引的叶子节点存主键值。如果通过二级索引查到主键后还要回到主键索引查询整行数据，这就是回表。如果查询字段都在二级索引里，就可以使用覆盖索引，避免回表。

联合索引要重点讲最左前缀原则。比如 `(user_id, status, created_at)`，查询条件从 `user_id` 开始连续命中时效果最好。索引失效常见于对字段使用函数、隐式类型转换、前置模糊匹配、范围条件后字段无法继续充分利用等。

最后可以结合慢 SQL 优化：先用 Explain 看访问类型、索引选择、扫描行数和 Extra，再结合业务查询条件调整索引。

## 3. 核心原理

B+ 树的非叶子节点只存索引键和指针，叶子节点存数据或主键值。这样单个页能容纳更多索引项，树高更低，磁盘 IO 更少。叶子节点之间通过链表连接，所以范围查询可以先定位起点，再顺序扫描。

主键索引也叫聚簇索引，数据按主键组织。二级索引不直接保存整行，而是保存主键，因此二级索引查询非索引字段时需要回表。覆盖索引就是让查询字段都命中索引，减少回表成本。

## 5. 易错点

不要只说“索引能加快查询”，还要补充它会降低写入效率、占用空间，并且不是所有查询都能命中。不要把最左前缀理解成“必须使用全部字段”，它强调的是从联合索引最左列开始连续使用。

回答慢 SQL 优化时，不要只说“加索引”，还要说明查询条件、排序、分页、回表和数据量分布。

## 6. 项目表达模板

在项目里，我会根据高频查询设计联合索引。比如用户维度的列表页，常见条件是 `user_id`、`status`，并按 `created_at` 排序，因此可以设计 `(user_id, status, created_at)`。优化时会先用 Explain 查看是否命中索引、扫描行数是否合理、是否出现 filesort，再判断是否需要覆盖索引或调整字段顺序。

## 7. 推荐训练

建议继续训练 B+ 树、MVCC、锁机制和慢 SQL 优化。索引题要练习从“数据结构”过渡到“业务查询设计”，这样更像真实项目经验。
