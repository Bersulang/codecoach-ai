# Spring 事务传播

## 1. 一句话解释

Spring 事务传播定义了一个带事务的方法调用另一个带事务的方法时，事务应该复用、挂起、新建还是嵌套。

## 2. 面试标准回答

面试时可以先说 `@Transactional` 的作用：它通过 Spring AOP 在方法执行前后开启、提交或回滚事务。事务传播解决的是方法之间相互调用时事务边界如何处理。

最常见的是 `REQUIRED`，如果当前有事务就加入，没有就新建。`REQUIRES_NEW` 会挂起当前事务并新建一个事务，适合主流程失败也希望保留的独立记录，比如操作日志。`NESTED` 是嵌套事务，依赖保存点，内层失败可以回滚到保存点，但外层事务仍可继续。

还要说明回滚规则。默认情况下，Spring 对运行时异常和 Error 回滚，对受检异常不回滚。如果希望受检异常也回滚，需要配置 `rollbackFor`。

最后补充事务失效场景：自调用、方法不是 public、异常被 catch 不抛出、类没有交给 Spring 管理、数据库引擎不支持事务等。

## 3. 核心原理

Spring 声明式事务依赖代理对象。外部调用代理对象时，事务拦截器会根据注解配置获取连接、开启事务、执行方法、提交或回滚。如果同类内部直接调用，就不会经过代理，事务增强也就不会触发。

事务传播本质上是对当前线程绑定的数据库连接和事务上下文进行管理。`REQUIRED` 复用上下文，`REQUIRES_NEW` 暂停旧上下文并创建新上下文，`NESTED` 在当前事务中创建保存点。

## 5. 易错点

不要只背传播行为名字，要用业务例子解释。比如 `REQUIRES_NEW` 适合独立提交日志，但也会增加连接占用。不要说加了 `@Transactional` 就一定回滚，异常类型、传播行为和代理调用都会影响结果。

另一个易错点是把事务边界放得过大。事务里包含远程调用、长时间循环或 AI 调用，会导致连接占用时间过长，影响并发能力。

## 6. 项目表达模板

在项目中，我通常把事务放在 Service 层，围绕一次完整业务写操作设置边界。比如创建订单时，订单、明细和库存扣减需要在同一个事务中使用 `REQUIRED`；但操作日志或失败记录如果希望独立保存，可以拆到单独 Service 并使用 `REQUIRES_NEW`。同时会避免在事务中执行耗时远程调用，防止数据库连接被长时间占用。

## 7. 推荐训练

建议继续训练 Spring AOP、数据库事务隔离级别、MySQL 锁机制和异常回滚。事务传播题要练习用真实业务例子解释不同传播行为，而不是只背枚举名。
