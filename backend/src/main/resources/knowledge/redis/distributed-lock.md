# Redis 分布式锁

## 1. 一句话解释

Redis 分布式锁是利用 Redis 的原子写入能力，在多个服务实例之间控制同一份共享资源只能被一个请求同时操作。

## 2. 面试标准回答

面试时可以先说使用场景：单机锁只能控制一个 JVM 内的并发，如果服务部署多实例，就需要分布式锁来控制跨进程并发，比如防止重复下单、库存扣减、任务重复执行。

Redis 分布式锁的基础实现是 `SET key value NX PX timeout`。`NX` 保证 key 不存在时才能写入，`PX` 设置过期时间，避免服务宕机后死锁。value 一般使用唯一标识，释放锁时要先校验 value，再用 Lua 脚本原子删除，防止误删别人的锁。

还要补充局限：锁超时时间不好设置，如果业务执行时间超过过期时间，锁可能提前释放。Redisson 通过看门狗机制自动续期，能缓解这个问题，但也不是强一致锁。RedLock 在工程中有争议，面试时可以说明它试图通过多 Redis 节点提升可靠性，但实现和运维复杂，很多业务更倾向用数据库唯一约束、MQ 串行化或 ZooKeeper 等方案。

## 3. 核心原理

Redis 锁依赖单条命令的原子性。加锁时必须把“判断不存在”和“写入锁”合并成一个原子操作，不能先 `GET` 再 `SET`。释放锁时也必须把“校验 value”和“删除 key”放到 Lua 脚本中，否则可能出现请求 A 的锁过期后，请求 B 加锁成功，请求 A 又误删 B 的锁。

锁过期时间是安全性和可用性的平衡。时间太短会提前释放，时间太长会导致异常时资源长时间不可用。看门狗续期能在业务未完成时延长锁时间，但也依赖客户端存活。

1. 为什么 Redis 分布式锁要设置 value？

3. 锁过期但业务还没执行完怎么办？

6. RedLock 是否一定可靠？

## 5. 易错点

不要只说 `SETNX`，因为旧式 `SETNX` 加 `EXPIRE` 如果不是原子操作，中间宕机会造成死锁。不要说“加了 Redis 锁就绝对安全”，它仍然有过期、网络抖动、主从切换和客户端停顿问题。也不要忽略幂等性，很多业务不能只靠锁，还要有数据库约束或状态机兜底。

## 6. 项目表达模板

在项目中，我使用 Redis 分布式锁控制跨实例并发，比如同一用户重复提交或同一资源重复处理。加锁使用 `SET NX PX`，value 使用请求唯一标识，释放时通过 Lua 脚本校验 value 后删除。对于执行时间可能较长的逻辑，会评估 Redisson 看门狗续期；但关键链路不会只依赖锁，还会用数据库唯一索引或业务幂等状态兜底，避免锁异常导致重复写入。

## 7. 推荐训练

建议继续训练幂等性、缓存一致性、MQ 重复消费和限流降级。分布式锁回答要突出“能解决什么、不能解决什么、项目里如何兜底”。
