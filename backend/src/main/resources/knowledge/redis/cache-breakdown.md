# Redis 缓存击穿

## 1. 一句话解释

缓存击穿是指某个热点 key 在过期瞬间被大量并发请求访问，缓存没有命中，请求集中打到数据库，导致数据库压力突然升高。

## 2. 面试标准回答

面试时可以先说定义：缓存击穿不是大量 key 同时失效，而是单个热点 key 失效。它通常发生在秒杀商品、热门活动配置、爆款内容详情等访问高度集中的场景。

接着说原因：热点 key 过期后，第一批并发请求同时发现缓存不存在，如果没有保护机制，就会一起查询数据库并回写缓存。

常见方案有两个。第一是互斥锁：只有一个请求拿到锁后查询数据库并重建缓存，其他请求等待或短暂重试。优点是实现简单，数据相对新；缺点是高并发下会有锁竞争，请求延迟会升高。第二是逻辑过期：缓存值里保存业务数据和逻辑过期时间，发现过期时先返回旧数据，再异步重建缓存。优点是抗并发能力强；缺点是短时间内可能返回旧数据。

最后结合项目场景表达：如果业务对实时性要求高，可以用互斥锁；如果更关注可用性和稳定性，比如首页热点内容、活动展示页，可以考虑逻辑过期。

## 3. 核心原理

缓存击穿的核心矛盾是“热点访问”和“缓存重建”同时发生。Redis 本身只负责缓存命中与过期，不会自动帮业务控制数据库回源并发，所以需要在应用层控制同一时间只有少量请求访问数据库。

互斥锁方案一般使用 Redis 的 `SET NX EX` 或本地锁加分布式锁。拿到锁的线程查询数据库并写回缓存，未拿到锁的线程可以 sleep 后重试，也可以快速失败。这里要注意锁过期时间不能太短，否则重建还没完成锁就释放，会出现多个线程同时回源。

逻辑过期方案不会让 key 真正过期，而是在 value 中放一个 expireTime。请求发现逻辑过期后，尝试加锁异步刷新缓存，同时当前请求仍返回旧数据。这种方案牺牲一点实时性，换取更平稳的流量。

## 4. 常见追问

1. 缓存击穿和缓存雪崩有什么区别？
2. 互斥锁方案下没有拿到锁的请求怎么办？
3. 逻辑过期会不会返回脏数据？
4. 锁过期时间应该怎么设置？
5. 如果数据库查询失败，缓存要不要回写？
6. 热点 key 如何识别？
7. 高并发下互斥锁会不会成为瓶颈？
8. 本地缓存能不能辅助解决缓存击穿？

## 5. 易错点

不要把缓存击穿说成“所有缓存都失效”，那是缓存雪崩。也不要只说“加锁就行”，还要补充锁竞争、锁超时、失败重试和数据库异常时的处理。回答逻辑过期时，不要回避旧数据问题，要说明它适合读多写少、对实时性要求没那么极端的场景。

另一个常见错误是只背方案，不讲取舍。面试官更关心你为什么在项目中选择互斥锁或逻辑过期，而不是能不能说出名词。

## 6. 项目表达模板

在我的项目里，某些活动或商品详情属于热点数据，如果缓存过期后大量用户同时访问，会直接打到数据库。我们针对这类 key 做了保护：对于实时性要求较高的数据，使用 Redis 互斥锁控制同一时间只有一个线程回源重建缓存；对于展示类热点数据，使用逻辑过期，先返回旧缓存，再异步刷新。这样既避免数据库被瞬时流量打满，也能根据业务容忍度平衡实时性和可用性。

## 7. 推荐训练

建议继续训练 Redis 缓存穿透、缓存雪崩、缓存一致性和分布式锁。训练时重点练习“场景 -> 问题 -> 方案 -> 取舍 -> 异常处理”的表达链路，不要只背方案名称。
