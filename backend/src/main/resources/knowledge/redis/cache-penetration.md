# Redis 缓存穿透

## 1. 一句话解释

缓存穿透是指请求查询的数据在缓存和数据库中都不存在，导致每次请求都绕过缓存打到数据库，严重时会被恶意流量利用。

## 2. 面试标准回答

面试时可以这样回答：缓存穿透的重点不是热点 key 过期，而是请求的数据本身不存在。比如用户传入一个不存在的商品 id、订单 id 或恶意构造的随机 id，缓存查不到，数据库也查不到。如果每次都不缓存空结果，那么同类请求会持续访问数据库。

解决思路通常有三层。第一层是参数校验，比如 id 格式、范围、权限和业务状态校验，明显非法的请求直接拦截。第二层是缓存空对象，当数据库查不到时，把空结果短时间写入 Redis，避免重复打库。第三层是布隆过滤器，把可能存在的 id 放入过滤器，请求先判断是否可能存在，不存在则直接拦截。

项目中要说明取舍：缓存空对象简单，但会占用缓存空间，也可能造成短时间数据不一致；布隆过滤器节省空间、适合海量数据，但有误判率，并且删除和更新维护更复杂。

## 3. 核心原理

缓存穿透的根源是“缓存没有记录不存在”。正常缓存只保存存在的数据，请求不存在的数据时，每次都会查数据库。缓存空对象就是把“不存在”也缓存起来，让 Redis 承担拦截作用。

布隆过滤器使用多个哈希函数和位图判断元素是否可能存在。它的特点是：判断不存在一定不存在，判断存在只是可能存在。因此它适合在数据库前面做快速过滤，但不能完全替代数据库查询。

在真实系统中，穿透往往和恶意请求有关，所以不能只依赖 Redis。还需要限流、鉴权、参数校验和监控告警配合，尤其是公开查询接口。

## 5. 易错点

不要把缓存穿透回答成“热点 key 过期”，那是缓存击穿。不要只说“用布隆过滤器”，还要说明误判、维护成本和适用场景。也不要忽略参数校验，很多穿透问题在进入缓存前就可以被拦截。

另一个易错点是缓存空对象时间设置过长。如果某条数据之后真的被创建，用户可能在空缓存过期前仍然查不到，所以通常要设置较短 TTL，或者在写入真实数据时主动删除空缓存。

## 6. 项目表达模板

在项目中，详情查询接口可能收到不存在的 id。为了避免这类请求反复访问数据库，我们先做参数和权限校验；如果数据库确实查不到，会把空结果以较短 TTL 写入 Redis。对于数据量较大且 id 集合相对明确的场景，可以在 Redis 前增加布隆过滤器，先判断请求 id 是否可能存在。这样既能挡住无效请求，也能避免空对象缓存无限膨胀。

## 7. 推荐训练

建议继续训练缓存击穿、缓存雪崩、布隆过滤器和接口限流。回答时重点区分“数据不存在”和“数据过期”，并练习缓存空对象与布隆过滤器的取舍。
